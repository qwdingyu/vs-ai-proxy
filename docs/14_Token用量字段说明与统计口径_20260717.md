# Token 用量字段说明与统计口径

日期：2026-07-17

## 本次结论

当前看板中的这些字段不是本地估算值，而是“上游响应里明确返回 usage 后，本代理记录并累计”的结果。

首页现在同时展示两类口径：

| 口径 | 用户要回答的问题 | 数据来源 |
| --- | --- | --- |
| 今日 / 本周 / 本月 | 最近自然日、ISO 周、自然月用了多少已知 token | `Statistics.period_usage.daily/weekly/monthly` |
| 最近 14 天趋势 | 哪几天消耗最高、覆盖率是否稳定 | `period_usage.daily` |
| 累计口径 | 当前代理生命周期或持久化快照内累计了多少已知 token | `Statistics` 累计字段和 `model_usage` |

周期统计同样遵守一个原则：请求数进入分母，只有上游返回 usage 的请求才贡献 token。也就是说，某一天显示 `有用量数据：1 / 2 次` 时，该日总 token 只来自 1 次已知用量，另一请求是未知，不是 0。

示例：

| 提供商 | 模型 | 请求 | 有用量数据 | 输入 Token | 输出 Token | 总 Token | 缓存命中的输入 | 模型内部思考用量 |
| --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| useai | UseAI - step-router-v1 -> step-router-v1 | 184 | 7 / 184 (4%) | 336,119 | 1,951 | 338,070 | 288,640 | 0 |
| kimi | kimi - kimi-for-coding -> kimi-for-coding | 73 | 3 / 73 (4%) | 129,640 | 929 | 130,569 | 85,849 | 0 |

这两行不能理解成“184 次请求一共用了 338,070 Token”。准确说法是：

- useai 共有 184 次可归属到该模型/上游的请求，其中只有 7 次服务商返回了可记录的用量；336,119 / 1,951 / 338,070 / 288,640 / 0 都只来自这 7 次有用量数据的请求。
- kimi 共有 73 次可归属到该模型/上游的请求，其中只有 3 次服务商返回了可记录的用量；129,640 / 929 / 130,569 / 85,849 / 0 都只来自这 3 次有用量数据的请求。
- 未返回用量的请求不会被当成 0 Token 累加；它们是“未知”，不是“免费”或“0 消耗”。

## 字段解释

### 请求

“请求”是该 provider + model + upstream 组合下的可统计请求数。源码中只要日志里 provider、model、upstream 任一字段不为空，就算作 token usage eligible request。

证据：

- `internal/store/store.go:307-317`
- `internal/store/store.go:335-337`

### 有用量数据

“有用量数据”是 `usage_reported_count / request_count`。

含义：

- 分母：这组模型一共有多少次请求进入统计范围。
- 分子：其中有多少次请求的服务商响应里带了用量，并被代理成功记录。
- 百分比：用量数据完整度，不是成功率，也不是上传到某个外部系统的比例。

例如 `7 / 184 次 (4%)` 表示：184 次请求中只有 7 次拿到了服务商返回的 token 用量。

常见原因：

- 服务商不返回用量。
- 流式响应没有包含最终用量帧。
- 请求在服务商返回用量前被客户端取消或断开。
- 某些协议转换或供应商兼容层没有透出用量。

本仓库不能仅凭汇总表判断具体哪一种原因，需要看对应日志和上游响应。

### 输入 Token

输入 Token 对应上游 usage 中的 `prompt_tokens`，也就是模型处理提示词、上下文、消息历史、工具定义等输入内容消耗的 token。

证据：

- `internal/proxy/token_usage.go:16-19`
- `internal/store/store.go:321-324`

### 输出 Token

输出 Token 对应上游 usage 中的 `completion_tokens`，也就是模型生成输出消耗的 token。

证据：

- `internal/proxy/token_usage.go:16-19`
- `internal/store/store.go:321-324`

### 总 Token

总 Token 对应上游 usage 中的 `total_tokens`。如果上游没有给 `total_tokens`，但给了输入和输出，代理会用 `prompt_tokens + completion_tokens` 补齐。

证据：

- `internal/store/store.go:343-354`
- `internal/provider/provider.go:120-141`

### 缓存输入

缓存输入对应 OpenAI 兼容 usage 里的 `prompt_tokens_details.cached_tokens`。

含义：

- 它是输入 Token 的子集。
- 它表示上游认为命中了提示词缓存的输入 token。
- 它已经包含在“输入 Token”和“总 Token”里，不能再次加到总 Token。

例如 useai 这一行：

- 输入 Token：336,119
- 缓存输入：288,640
- 正确理解：336,119 个输入 token 中，288,640 个被上游标记为缓存命中。
- 错误理解：338,070 + 288,640。

证据：

- `internal/proxy/token_usage.go:22-24`
- `internal/proxy/metrics.go:46`
- `web/index_html_test.go:64-65`

### 推理输出

推理输出对应 OpenAI 兼容 usage 里的 `completion_tokens_details.reasoning_tokens`。

含义：

- 它是输出 Token 的子集。
- 它表示上游把多少输出 token 归类为 reasoning / thinking token。
- 它已经包含在“输出 Token”和“总 Token”里，不能再次加到总 Token。
- 当前表中为 0，表示已返回的用量数据中没有记录到 reasoning_tokens，或者服务商返回为 0；不能推出模型完全没有内部推理。

证据：

- `internal/proxy/token_usage.go:25-27`
- `internal/proxy/metrics.go:47`
- `web/index_html_test.go:64-65`

## 当前文案风险

“已上报”对普通用户容易误解为“已经上报到云端”或“请求成功上报”。更清晰的产品文案建议：

| 当前文案 | 建议文案 | 原因 |
| --- | --- | --- |
| 已上报 / Token 明细覆盖 | 有用量数据 | 普通用户更容易理解为“多少次请求拿到了可统计用量” |
| 缓存输入 | 缓存命中的输入 | 明确它属于输入 Token 子集 |
| 推理输出 / 推理输出子项 | 模型内部思考用量 | 比“子项”更通俗，同时说明它不是额外相加项 |

建议在 UI 上加一句短说明：

> 只统计服务商返回的用量；未返回的不估算、不按 0 计算。缓存命中的输入、模型内部思考用量是明细，已包含在总 Token 中，不重复相加。

## 审核结论

- 当前核心统计口径是正确的：未知用量不被当成 0。
- 当前字段文案已改为“有用量数据”，比“已上报”更直观。
- 缓存输入和推理输出是子集，不是额外成本项。
- 在有用量数据比例只有 4% 的情况下，不能把表中总 Token 当成真实总消耗，只能当成“已返回用量部分的消耗”。
