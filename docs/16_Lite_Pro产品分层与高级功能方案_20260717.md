# Lite / Pro 产品分层与高级功能方案

日期：2026-07-17

> 2026-07-18 纠偏：本文是高级功能候选清单，不是拆仓或立即商业化的批准依据。当前最优策略是保持一个公开仓库和一个产品，先验证真实付费意愿与高级能力闭环。涉及 Lite / Pro、拆仓、授权和官方 Copilot 第三方模型开放风险时，以 `32_产品定位与仓库策略_20260718.md` 为准。

## 产品定位判断

当前产品最有价值的位置不是“又一个 API 转发器”，而是：

> 面向 AI 编程工具和桌面开发环境的本地/团队级 LLM 接入治理层。

用户真正痛点：

- 不同 provider 的 OpenAI 兼容程度不同。
- 模型、base_url、参数、流式、工具调用、usage 返回经常不一致。
- 出错时普通用户看不懂 400/404/499/502/context canceled。
- 长会话会膨胀，客户端经常提前断开。
- 大家需要知道哪个模型稳定、哪个便宜、哪个适合 coding。
- 需要一个稳定的对外模型名，而不是每次手动换 provider/model。

因此护城河不在“支持更多 provider”本身，而在：

- 兼容治理。
- 可解释诊断。
- 健康感知路由。
- 成本与 token 可观测。
- 面向 VS/Copilot/Codex/Kimi 等真实客户端的专项适配。

## Lite 版本

Lite 应该做“基础功能稳定、低心智负担”。

建议包含：

- OpenAI 兼容代理。
- Provider 配置。
- 模型配置。
- 基础请求日志。
- 基础错误解释。
- 基础 token 统计。
- 手动模型测试。
- 配置导入导出。
- 单机运行和 Windows exe。

Lite 不建议包含：

- 多租户计费。
- 账号池商业化。
- 复杂权重路由。
- 后台自动探测。
- 高级成本优化。
- 自动模型分层策略。

Lite 的目标：

- 用户能配得上。
- 出错能看懂。
- 本地测试稳定。
- 不制造新复杂度。

## Pro 版本

Pro 应该提供“高级治理能力”，让重度用户和小团队愿意付费或持续使用。

### Pro 1：智能 Auto Router

核心价值：

- 对外只暴露稳定模型名，例如 `auto`、`coding`、`coding-fast`、`coding-pro`。
- 对内根据健康、延迟、成本、能力选择真实 provider/model。

能力边界：

- 不是简单 fallback。
- 不是无条件轮询。
- 必须可解释：每次请求为什么选这个，为什么跳过其他。

建议策略：

- `稳定优先`：成功率和错误率优先。
- `速度优先`：TTFT 和总耗时优先。
- `成本优先`：低成本 provider 优先。
- `平衡模式`：成本、速度、成功率综合。

低成本实现路径：

- P1 先做 priority + cooldown + last_used。
- P2 再做 sticky session。
- P3 再做 top-K 加权。

### Pro 2：Provider 健康中心

核心价值：

- 用户不需要猜“是我配置错了，还是上游坏了”。

功能：

- provider/model 健康状态。
- 最近错误分类。
- Token usage 覆盖率。
- 流式稳定性。
- 首 token 时间。
- 499/超时/上下文过大提示。
- 一键重新探测。

从 API-Switch 吸收：

- 手动测速。
- 同渠道串行探测。
- 记录响应延迟。

从 sub2api 吸收：

- 周期性 monitor。
- 历史状态。
- degraded / failed / operational 状态。

必须避免：

- URL 可达就当模型可用。
- 探测失败直接覆盖用户手动 enabled。
- 无并发预算地批量探测。

### Pro 3：参数兼容治理

核心价值：

- 这是当前产品最接近护城河的能力。很多 400/404/502，本质不是模型坏，而是参数、URL、协议、客户端行为不兼容。

功能：

- base_url 规范化，例如避免 `/v4/v1/chat/completions`。
- provider 参数白名单/黑名单。
- 不兼容参数自动剥离或降级。
- 工具调用格式治理。
- 空 assistant message 修复或提示。
- stream_options.include_usage 按 provider 能力启用。
- reasoning/thinking 参数按 provider profile 控制。

Lite 可以有基础规则；Pro 提供：

- provider profile。
- 规则解释。
- 自定义规则。
- 变更前后 diff。
- 兼容性测试套件。

### Pro 4：Token 与成本治理

核心价值：

- 用户会关心“钱花在哪里”和“为什么上下文突然变大”。

功能：

- 用“有用量数据”解释 token usage 覆盖率。
- 服务商返回 token 与估算 token 分开展示。
- 缓存命中的输入、模型内部思考用量解释。
- 按 provider/model/session 统计。
- 大上下文预警。
- 历史膨胀提示。
- 成本估算。
- 低覆盖率 provider 标红。

从 sub2api 吸收：

- 更细的 usage log。
- 缓存 token 字段。
- 按模型/账号/用户维度聚合的思想。

当前产品不建议立刻做：

- 余额系统。
- 扣费系统。
- 支付系统。

### Pro 5：长会话治理

核心价值：

- VS/Copilot/Codex 类客户端经常有长 session、工具调用、大上下文、流式取消。普通代理不会处理这些问题。

功能：

- session 维度统计。
- 大 session 告警。
- 新建 session 建议。
- sticky session 到同一 provider/model。
- session 污染诊断。
- 工具调用完整性检查。
- 499/context canceled 友好解释。

### Pro 6：可解释错误诊断

核心价值：

- 错误文案是产品体验的一部分。用户不想看一大坨日志。

建议错误文案分层：

- 一句话结论。
- 用户可操作建议。
- 技术详情折叠。
- 证据字段：状态码、provider、model、request size、stream state、upstream body 摘要。

示例：

当前类似：

> 上游拒绝参数/模型，检查模型名、provider 绑定、base_url 和不兼容参数治理。

建议：

> 上游不接受这次请求。优先检查模型名和参数兼容性；如果只在旧会话出现，建议新建会话后重试。

技术详情折叠里再放：

- HTTP 400/404。
- upstream body。
- request size。
- provider/model。
- stream state。

## 最优路线图

### 第一阶段：修正基础认知和文案

目标：先让用户看懂，不误判。

TODO：

- “已上报 / Token 明细覆盖”改为“有用量数据”。
- “缓存输入”改为“缓存命中的输入”。
- “推理输出 / 推理输出子项”改为“模型内部思考用量”，并说明已包含在输出 Token。
- 错误文案简化，一句话结论优先。
- 日志详情保留技术证据。

### 第二阶段：正确的 auto 模型

目标：对外稳定，对内透明。

TODO：

- 定义逻辑模型：auto、coding-fast、coding-pro。
- 统一候选过滤。
- 记录 requested_model 和 upstream_model。
- 明确不对 unknown model 默认静默 fallback。
- 每次路由产出 route decision。

### 第三阶段：轻量健康感知路由

目标：少量代码带来明显稳定性。

TODO：

- 引入 provider/model health state。
- 可重试错误进入 cooldown。
- 不可重试错误不污染健康。
- cooldown 到期单探针恢复。
- priority + last_used 做轻量轮换。

### 第四阶段：Pro 高级治理

目标：形成差异化。

TODO：

- 后台健康探测。
- sticky session。
- 成本/速度/稳定策略。
- top-K 加权选择。
- token 覆盖率与成本估算。
- provider compatibility profile。

## Lite / Pro 功能边界

| 功能 | Lite | Pro |
| --- | --- | --- |
| OpenAI 兼容代理 | 是 | 是 |
| Provider 配置 | 是 | 是 |
| 基础日志 | 是 | 是 |
| 基础 token 统计 | 是 | 是 |
| Token 覆盖率解释 | 是 | 是 |
| 手动模型测试 | 是 | 是 |
| 稳定逻辑模型名 | 基础 auto | 多策略 auto |
| 健康感知路由 | 简单 cooldown | 健康评分 + 半开探针 |
| 后台健康探测 | 否 | 是 |
| Sticky session | 否 | 是 |
| 成本估算 | 否或基础 | 是 |
| 参数兼容 profile | 基础内置 | 可配置、可解释 |
| 高级诊断 | 基础文案 | 诊断中心 |
| 多租户余额/支付 | 否 | 暂不建议 |

## 风险控制

必须坚持的原则：

- 不把所有失败都当成渠道坏。
- 不在流式响应开始后透明重放。
- 不把未返回用量当 0。
- 不让后台探测覆盖用户手动启停。
- 不引入 sub2api 式多租户复杂度，除非产品定位明确升级为 SaaS 网关。
- 不把“auto”做成黑盒；必须可解释。

## 最终建议

最优方案是：

1. Lite 保持清晰、稳定、低配置成本。
2. Pro 做“智能 auto + 健康探测 + 可解释诊断 + token/成本治理”。
3. 从 API-Switch 吸收逻辑模型和人工测速的边界，但修正其 auto 过滤和伪轮换问题。
4. 从 sub2api 吸收 sticky、load-aware、temp unschedulable、monitor、usage 维度，但不搬多租户计费体系。

这样形成的护城河不是“支持多少模型”，而是“在真实 AI 编程工具里稳定、可解释、低成本地把模型用起来”。
