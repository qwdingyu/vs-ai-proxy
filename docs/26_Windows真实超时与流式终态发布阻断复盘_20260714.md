# Windows 真实超时与流式终态发布阻断复盘（2026-07-14）

## 1. 背景

Windows 真实运行日志出现以下组合：

```text
request_bytes ≈ 1007.6 KB
upstream_bytes ≈ 1.01 MB
effective timeout = 90s
stream_state = upstream_connecting
context deadline exceeded
HTTP 502
```

同一阶段刚完成一次较大的 Copilot 通用工具协议改造。发布前必须分别回答两个问题：

1. 这次 Windows 超时发生在哪个阶段，代理还能提供什么有效诊断；
2. 通用工具改造是否破坏普通聊天、流式终态或 OpenAI/Ollama 兼容。

不能把二者混成“工具未注册”，也不能因本机 macOS 无法复现就否定 Windows 现场。

## 2. 原因与证据边界

### 2.1 Windows 日志已经证明的内容

- 客户端请求和代理治理后的上游请求都约为 1 MiB，没有数量级膨胀。
- 请求在 90 秒有效预算内没有取得可读上游响应流。
- `upstream_connecting` 在调用 `Provider.ChatStream` 前设置，响应工具归一化尚未执行。
- 本次请求没有可供代理校验或转换的完整 `tool_calls`。

### 2.2 Windows 日志不能证明的内容

- 不能证明工具没有注册或工具 schema 被代理删除。
- 不能证明上游已经返回完整工具调用后被代理丢失。
- 不能仅凭 1 MiB 字节数判断超过模型 token 上限。
- 不能只凭旧的 `upstream_connecting` 区分 DNS、TCP、TLS、请求上传或等待响应头。

最符合现有证据的解释仍是：请求在有效预算内没有收到上游响应头或首个 SSE 字节。真正的上游性能根因仍需新的 Windows 日志或真实渠道重放确认。

## 3. 排查过程

1. 以约 1 MiB 正文启动本地可控 HTTP 上游；上游完整读取请求，但故意不返回响应头。
2. 稳定复现 `context deadline exceeded`，证明该形态与 Windows 日志一致。
3. 审查 `be2cfa6` 全部生产差异，按普通聊天、OpenAI raw/typed/SSE、Ollama 双向转换、fallback、配置和平台构建分组。
4. 检查测试是否复用生产解析器，发现模拟 VS 客户端的解码器与生产代码同源，存在假阳性风险。
5. 使用保持 transport body 打开的 reader 验证应用层 `[DONE]` 和传输层 EOF 的边界。
6. 使用标准多行 SSE、refusal-only 响应和 `functions:[]/null` 构造永久回归。

修复前稳定复现：

- 已收到 `[DONE]` 仍等待 EOF；
- 多行空终态在整体校验前写给下游；
- 合法 refusal-only 响应被当成空成功拒绝；
- `functions:[]/null` 被误判为工具请求，改变普通聊天默认值；
- 超时错误只能看到笼统的 `upstream_connecting`。

## 4. 根因

### 4.1 `[DONE]` 与 EOF 混淆

工具尾部和成功终态改为校验后提交，但读取循环仍以 `scanner.Scan()` 读到 EOF 才结束。部分网关发送标准 `[DONE]` 后会延迟关闭 body，导致代理已经拥有完整响应却迟迟不向 VS 提交终态。

### 4.2 多行 SSE 终态识别按物理行解析

逻辑事件已把多条 `data:` 拼为一段 JSON，但终态检测再次按换行拆开，无法识别跨行的 `finish_reason`，从而绕过终态缓存门禁。

### 4.3 payload 定义遗漏 refusal

OpenAI 的 refusal-only 响应可以没有 `content`，但 `refusal` 本身是合法且应展示的结果。raw、typed 和 direct SSE 的“是否有有效响应”判断只认 content/reasoning/tools，造成正常拒绝响应被误报为空。

### 4.4 legacy functions 判断原始字节长度

`len(req.Extra["functions"]) > 0` 判断的是 JSON 文本字节数。`[]` 和 `null` 都有字节，却没有声明任何函数，普通聊天因此错误跳过 `temperature=0.7` 和 `max_tokens=4096` 的原有默认。

### 4.5 测试客户端复用生产聚合器

工具 E2E 的模拟客户端复用了生产 parser/merge/build 函数。若生产端和测试端同时以同一种错误方式拼接 name、id 或 arguments，测试仍可能通过。

## 5. 解决方案

### 5.1 统一应用层终态

- `openAIStreamEventProcessor` 作为 direct SSE 和 OpenAI/Ollama 转换共享门禁。
- 收到并完成 `[DONE]` 逻辑事件后立即退出读取循环，不等待上游关闭 transport body。
- 即使上游只发送 `data: [DONE]\n` 而没有额外空行，也按应用层终态结算。
- 工具尾部、finish 和 `[DONE]` 仍在完整性校验后原子提交。
- 多行 SSE 终态按逻辑事件正文解析，只识别行首 `data:` 字段，不受 `id:`、`event:` 或注释值中的 `data:` 干扰。
- 对上游省略空行但已确认完整的事件，代理补齐标准 `\n\n` 分隔，避免 finish 和 `[DONE]` 被严格客户端合并。

### 5.2 refusal 全链路保真

- `provider.Message` 明确保存标准 `refusal` 字段。
- raw、typed、provider SSE collector 和 direct SSE 都把非空 refusal 视为有效 payload。
- stream-to-nonstream fallback 重新合成 SSE 时输出 `delta.refusal`。
- OpenAI 转 Ollama 时，在没有 content 的情况下把 refusal 作为可见 content 返回，避免静默空响应。

### 5.3 保持普通聊天原行为

legacy `functions` 必须成功解析为非空数组才视为工具请求。`[]`、`null` 和无效形态不再关闭普通聊天默认值；非空 functions 和 modern tools 仍避免代理猜测工具输出上限。

### 5.4 Windows 跨平台阶段诊断

使用 Go 标准库 `net/http/httptrace`，只记录阶段，不记录 URL、Header 或正文：

| `upstream_stage`（当前 HTTP hop） | 含义 |
| --- | --- |
| `resolving_dns` | 正在解析 DNS |
| `connecting` | 正在建立 TCP/代理连接 |
| `tls_handshake` | 正在进行 TLS 握手 |
| `writing_request` | 已有连接，正在写请求头或正文 |
| `waiting_response_headers` | 请求已写完，等待上游响应头/首字节 |
| `receiving_response_headers` | 已收到响应头首字节，正在完成响应建立 |
| `preparing_request` | 尚未进入可观测网络阶段，可能是请求构造或极短预算 |

阶段状态在同一 HTTP hop 内只前进不回退；发生重定向时从 `preparing_request` 重新开始，避免并发 trace 回调或前一跳状态产生误导。该诊断适用于 Windows 和 macOS，不改变超时、路由或重试策略。

### 5.5 独立 wire 测试

模拟 VS 客户端的测试解码器只使用 `encoding/json` 读取实际 SSE wire，不再调用生产 parser、merge 或 normalizer。生产低层函数仍保留各自单元测试，两类证据不再同源。

## 6. 正式回归测试

新增并永久保留：

1. 上游发送 `data: [DONE]\n`（有无空行）后保持 body 打开，代理必须在测试时限内返回并提交 `[DONE]`。
2. 多行 SSE 的空 `stop` 必须失败，且不得提前暴露 finish 或 `[DONE]`。
3. raw、typed fallback、direct SSE refusal-only 响应必须成功并保留 refusal。
4. provider SSE refusal 分片必须聚合并通过 JSON round-trip。
5. `functions:[]` 和 `functions:null` 必须保留普通聊天默认值。
6. 约 1 MiB 请求已写完但上游不返回响应头时，错误必须包含 `upstream_stage=waiting_response_headers`。
7. 任意 Copilot 工具 E2E 使用独立 wire 解码器验证 id/type/name/arguments/finish reason。
8. provider 重试直接比较每次 RoundTrip 的 context deadline，禁止用易受 CI 调度影响的短 wall-clock 阈值冒充共享预算证明。

发布门禁仍为：

```bash
make tool-check
go test ./... -count=1
go test -race ./... -count=1
go vet ./...
make i18n-check
make release-check
```

并显式构建 Windows x64、macOS x64、macOS arm64。项目不使用 Docker，不能用容器结果替代桌面平台构建。

## 7. Windows 复测建议

1. 先记录 `vs-ai-proxy.exe --version` 和 request ID。
2. 分别用新 session 小请求、旧 session 约 1 MiB 请求测试同一模型和工具。
3. 若错误为 `upstream_stage=waiting_response_headers`，说明请求已写完，重点查 useai2/new-api 渠道排队、模型首响应和大上下文能力。
4. 若停在 `resolving_dns/connecting/tls_handshake`，重点查 Windows DNS、系统代理、安全软件、证书链和 CDN 网络。
5. 若停在 `writing_request`，重点查上传链路、系统代理/WAF 和上游是否提前关闭连接。
6. 若已经 HTTP 200 且 `response_tools` 有工具但 VS 不执行，再保存脱敏 SSE 检查客户端协议；不要继续按工具名补丁。

## 8. 剩余风险

- 本地测试不能代替真实 Windows `useai2/gpt-5.6-sol` 付费重放。
- 代理无法迫使上游在 90 秒内返回首字节，也没有证据支持无限延长 VS 等待预算。
- `httptrace` 能定位到网络生命周期阶段，但不能直接解释上游网关内部排队、渠道选择或模型推理耗时。
- 真实 VS/.NET SDK 版本差异仍需 Windows 客户端复测；当前独立 wire 解码器验证协议结构，不冒充官方 SDK。

## 9. 维护建议

- `[DONE]` 是 SSE 应用层终态，不能再次改成等待 EOF 后提交。
- 修改 payload 判定时同步检查 content、reasoning、refusal、modern/legacy tools 和截断终态。
- 修改工具流时同时维护 raw、typed、direct SSE、Ollama 双向转换和独立 wire E2E。
- 新的 Windows 现场日志必须保留版本、request ID、`upstream_stage`、request/upstream bytes 和 stream state。
- 未取得原始上游响应前，不得把超时写成“工具未注册”或“代理丢失工具”。
