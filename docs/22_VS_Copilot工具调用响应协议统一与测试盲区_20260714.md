# VS Copilot 工具调用响应协议统一与测试盲区（2026-07-14）

## 1. 背景

`v0.2.51` 发布后，正式测试、race、vet 和 Windows CI 均能通过，但继续审查工具调用异常路径时，使用临时 Go overlay 复现出了多项稳定失败：工具调用结构存在，代理却返回 `finish_reason=stop`；SSE 错误帧和残缺参数被当成成功；Ollama 非流式响应丢失工具等。

审查时创建过 `/tmp/vs_ai_proxy_review_proxy_test.go` 和 `/tmp/vs_ai_proxy_review_provider_test.go`，确认问题后又删除。这个做法不正确：这些场景可重复、涉及核心功能，并且现有门禁没有覆盖，理应直接沉淀为仓库正式回归测试，而不是下次重新编写。

本次修复遵循两个约束：

1. 不按 provider、模型名或工具名逐个加条件。
2. 不引入新的依赖或庞大框架，只统一现有 OpenAI SSE 解析边界和工具响应归一化边界。

## 2. 证据边界

### 2.1 已经确定的事实

- 临时契约测试能稳定复现协议缺陷，不依赖真实网络或随机时序。
- `make tool-check` 当时通过，说明发布门禁只覆盖了正常路径，不能证明错误帧、截断和 object arguments 安全。
- 原始测试中有一条曾通过，是因为测试先反序列化到 `provider.FunctionCall`；该类型会把 object arguments 自动转成字符串，掩盖了 raw wire JSON 仍是 object 的事实。改为直接检查 JSON wire 类型后稳定失败。

### 2.2 不能写成定论的内容

审查时本机 `logs.json` 的最近记录没有同时包含可用的 `request_tools` / `response_tools` 样本，因此不能把用户某一次真实“无法运行工具”武断归因到某个 provider 或本次某一条 bug。

本次能确认的是：这些协议缺陷足以导致工具调用失败；修复后可以消除代理自身的确定性问题。真实上游若没有完整返回工具参数、客户端提前断开或模型根本没有产生工具调用，代理仍不能凭空补造。

## 3. 根因

### 3.1 SSE 聚合存在两套实现

provider 的 `stream=false -> SSE` 兜底只聚合文本；proxy 的聚合器另行处理工具分片。两套实现对 BOM、错误帧、legacy `function_call`、object arguments、结束原因和截断的判断不一致。

后果：同一份上游响应经不同入口可能得到不同结果，修复其中一个入口不能形成闭环。

### 3.2 direct SSE 只有逐行透传，没有工具终态

原 direct SSE 路径会把每行直接写给下游；辅助 accumulator 的解析错误被忽略，EOF 也不校验工具参数是否完整。

后果：

- `data: {"error":...}` 可以作为 HTTP 200 流的一部分被转发。
- `{"path":` 之类残缺参数遇到 EOF 后，handler 仍返回成功。
- 已写出字节后的协议错误还会被记录成 provider 成功，污染健康排序。

### 3.3 工具参数和结束原因没有统一语义

不同入口分别处理 arguments：

- typed `FunctionCall.UnmarshalJSON` 能处理 object。
- raw JSON 原样保留 object。
- SSE 合并曾使用 `fmt.Sprint(map)`，产生 `map[path:a.txt]`，不是 JSON。
- Ollama converter 原样复制 object。

同时，完整 `tool_calls` 可能保留上游错误的 `stop`；而 `length/content_filter` 下的残缺调用又可能继续暴露给客户端执行。

### 3.4 DSML 探测按物理行计数

标准 SSE 事件之间通常有空行。旧实现最多探测 4 个物理行，因此 role 和 reasoning 两个事件之后才出现的 DSML 会直接漏过。

命中 DSML 后，旧实现先把剩余流无界写入 `bytes.Buffer`，再进入 64 MiB 聚合限制，限制生效太晚。

### 3.5 Ollama 非流式只读取文本

`OllamaProvider.Chat` 只读取 `message.content/thinking`，忽略 `message.tool_calls` 和 `done_reason`，然后固定返回 `finish_reason=stop`。

### 3.6 direct SSE 把事件误当成物理行

第一轮统一后，provider collector 已支持标准多行 SSE，但 direct stream 仍逐行调用 JSON 解析器。复查时用正式契约稳定复现：同一个事件包含多条 `data:` 时，第一条不完整 JSON 会被立即拒绝；`event: error` 与下一条 `data:` 也无法关联，错误事件会被当成普通输出透传。

同时，8 KiB DSML probe 之后才出现的工具调用会边解析边写出；如果最终以 `length/content_filter` 结束，客户端已经看到了可执行 `tool_calls`，EOF 校验无法撤回已发送的内容。

### 3.7 typed fallback 与 Ollama 流存在 silent success

主 raw JSON 路径已经校验残缺工具参数，但 stream-to-nonstream fallback 会直接调用 `Provider.Chat`。OpenAI/Ollama typed 响应当时没有相同校验，因此可能绕过 raw 边界。

Ollama NDJSON 转 OpenAI SSE 时，畸形 JSON 行原先会被 `continue` 吞掉，循环结束后仍补发 `[DONE]`。工具行损坏时，用户只会看到“没有工具调用”，而不是明确的上游协议错误。

## 4. 排查过程

1. 先运行现有 `make tool-check`、定向测试、race 和 vet，确认正常路径没有已有红灯。
2. 从 raw JSON、direct SSE、SSE 聚合、fallback、DSML、Ollama 六个入口构造同一组工具协议场景。
3. 第一轮正式 proxy 契约测试结果为 1 项通过、8 项失败；唯一通过项是 typed 反序列化造成的假阳性。
4. 将假阳性改为 wire JSON 断言后，object arguments 也稳定失败。
5. 先修共享解析/归一化边界，再逐入口接入；每次出现旧测试失败，都判断是兼容回归还是旧测试固化错误语义。
6. 发现 provider/proxy 各自新增 SSE 状态机后，停止继续叠加，改为由 provider 提供唯一标准 OpenAI SSE collector，proxy 只叠加 VS/DSML 业务规则。

## 5. 解决方案

### 5.1 唯一标准 SSE collector

`internal/provider` 的 `CollectOpenAIChatSSE` 统一负责：

- UTF-8 BOM。
- 标准 SSE event framing 和多行 `data:`。
- 兼容缺少事件空行的 OpenAI-compatible 网关。
- 顶层 `error` 和 `event: error`。
- 总响应字节上限。
- modern `tool_calls` 分片。
- legacy `function_call` 分片。
- object arguments 转规范 JSON string。
- `[DONE]` / `finish_reason` 终态校验。
- 工具参数完整性与截断语义。

proxy 的非流式 SSE 聚合、nonstream-to-stream fallback 和 DSML 聚合均复用该 collector，不再维护第二套标准 SSE 聚合算法。

### 5.2 唯一工具响应归一化边界

`internal/proxy/tool_response_normalizer.go` 统一负责：

- 工具名按大小写不敏感匹配请求声明，但输出请求声明的精确拼写。
- object arguments 用 `encoding/json` 编码为 JSON string。
- 完整 modern 工具调用统一结束为 `tool_calls`。
- 完整 legacy 调用统一结束为 `function_call`。
- `length/content_filter` 下不暴露任何可执行工具调用。
- raw HTTP 200 顶层 `error` 作为失败，不再返回 200 成功。

标准 OpenAI 未声明工具仍沿用 stable 透传策略，避免代理误删 Copilot 已知工具；DSML 和别名转换仍只能落到当前请求真实声明的工具，不扩权。

### 5.3 direct SSE 状态校验

direct SSE 按完整 SSE 事件而不是物理行处理，兼容标准多行 `data:` 和缺少空行的 OpenAI-compatible 输出，并能关联 `event: error` 与事件正文。

普通内容事件仍即时透传，不增加整段响应延迟。一旦任意位置出现 modern/legacy 工具调用，只缓存从该事件开始的“工具尾部”；终态、参数和结束原因全部校验通过后才输出。工具尾部与单事件缓存均受 64 MiB 上限保护，因此 late tool、截断和多行大事件不会绕过安全边界或造成无界内存增长。

modern `tool_calls` 统一结束为 `tool_calls`；legacy `function_call` 保留旧结构并结束为 `function_call`。两者不再共用一个无法区分协议类型的结束状态。

如果错误发生在普通内容已经写给下游之后，代理不能切换 provider 或改写已经发送的字节，但必须：

- 返回错误而不是成功。
- 将 provider 记录为失败而不是成功。
- 不启动跨 provider 混流。

### 5.4 DSML 探测和内存上限

- 探测条件从“4 个物理行”改为“最多 8 KiB 前缀”，空行不再提前耗尽探测窗口。
- 命中 DSML 或显式工具调用后，继续读取受 64 MiB 总量限制保护。
- 该限制有小容量单元测试，不依赖分配真实 64 MiB 数据验证。

8 KiB 是首 token 延迟与方言识别之间的明确权衡：如果 DSML 在超过 8 KiB 普通响应之后才首次出现，代理不会为了方言兼容无限期缓存所有普通 SSE。遇到真实样本时，应先保存脱敏响应 fixture，再基于证据调整，而不是猜测模型名特判。

## 6. 终态规则

| 上游结果 | 下游行为 |
| --- | --- |
| 完整 `tool_calls` + `stop/空/私有 finish` | 输出工具调用，结束原因统一为 `tool_calls` |
| 完整 legacy `function_call` | 保留 legacy 结构，结束原因统一为 `function_call` |
| 残缺工具 + `stop` 或无完成标记 | 协议错误，不作为成功 |
| 任意工具 + `length/content_filter` | 保留截断原因，不暴露可执行工具 |
| HTTP 200 + 顶层 `error` | provider/候选失败 |
| SSE `error` event | provider/候选失败 |
| object arguments | 输出规范 JSON string |
| 未声明标准 OpenAI 工具 | stable 模式原样保留，不做别名扩权 |
| 未声明 DSML 工具 | 保留普通文本，不转换执行 |
| direct 多行 SSE | 按事件合并后校验，输出合法 SSE 事件 |
| late tool + `length/content_filter` | 已输出的普通内容保留，工具尾部不发送并记录 provider 失败 |
| 畸形 Ollama NDJSON | 返回协议错误，不补发虚假的 `[DONE]` |

修改工具协议后至少执行：

```bash

```

还必须构建 Windows x64、macOS x64、macOS arm64，并做当前 macOS 原生可执行文件冒烟。项目不使用 Docker，不能用容器测试替代 Windows/macOS 可执行文件验证。

## 9. 建议

1. 新增上游方言时，先判断能否归入标准 collector 或统一 normalizer，禁止先在 handler 中加模型/provider 特判。
2. 工具失败排查必须同时看 `request_tools`、`response_tools`、`finish_reason`、`stream_state` 和原始上游终态。
3. 没有脱敏响应 fixture 时，只能写“可能原因”，不能把 provider/model 猜测写成根因。
4. 代理不能修复上游根本没有返回的参数，也不能在下游流开始后安全切 provider。
5. 发布前以正式仓库测试和原生平台产物为准，不依赖 `/tmp` overlay 或 Docker。
