# 近期稳定性修复与踩坑总复盘

日期：2026-07-23

## 背景

用户在真实 VS/Copilot 工具调用场景中观察到 UseAI `deepseek-v4-flash` 间歇性失败，日志中出现过：

```text
POST /v1/chat/completions - 502
error_code=network_error
reason="无法连接上游"
request_bytes="400 KB 级别"
upstream_bytes="400 KB 级别"
```

同时，官方 DeepSeek 或其他真实上游也可能出现 `upstream_stream_interrupted`、`client_gone`、`client_deadline_reached` 等现象。这里必须明确：代理程序无法从根本上杜绝公网网络波动、上游网关断流、上游算力排队、客户端主动取消或客户端等待超时；本轮修复的目标是减少误报、避免危险重放、提高观测能力，并让发布前测试可以复现和量化间歇性问题。

## 本轮核心原则

1. 不大改核心转发架构。
2. 不引入新依赖。
3. 不把观测逻辑接入路由、重试或 fallback 决策。
4. 不把所有失败笼统归因成“无法连接上游”。
5. 不在无法确认安全的阶段重放非幂等 chat POST。
6. 对真实上游问题只做准确暴露和低风险缓解，不承诺完全消除。

## 已做改动

### 1. provider 层记录结构化上游尝试

文件：

- `internal/provider/provider.go`
- `internal/provider/provider_test.go`

新增 `UpstreamAttempt`，用于描述 provider 内部一次真实 HTTP 尝试：

- `Stage`：来自 `httptrace` 的最后网络阶段。
- `Elapsed`：单次上游尝试耗时。
- `HTTPStatus`：上游明确返回 HTTP 响应时的状态码。
- `Error`：脱敏后的错误文本。

新增内部错误包装：

- `upstreamAttemptsError`
- `upstreamTransportError`
- `UpstreamAttempts(err error)`

目的：

- 区分“请求还没提交”和“请求已经写出但上游没响应”。
- 让 proxy 层能通过 `errors.As` 读取结构化尝试，不依赖具体 provider 类型。
- 日志不保存 URL、Header、请求体、响应体，避免泄露 API Key 或用户提示词。

### 2. provider 手动重试边界收紧

文件：

- `internal/provider/provider.go`

重试边界按 HTTP 阶段收敛：

- 允许短重试：`preparing_request`、`resolving_dns`、`connecting`、`tls_handshake`。
- 不允许代理手动重试：`writing_request`、`waiting_response_headers`、`receiving_response_headers`。
- `context.Canceled` 和 `context.DeadlineExceeded` 不重试。
- HTTP 5xx 在 `DefenseEnabled` 下保留历史短重试，用于 new-api/sub2api 单渠道 503 直接透出场景。
- HTTP 4xx、429 不做盲目重试。

关键取舍：

- chat completions 是非幂等 POST。进入写请求或等待响应头后，上游可能已经收到请求，再由代理手动重放会带来重复计费、重复生成、重复工具调用风险。
- 陈旧空闲连接等 Go 标准库能确认安全的场景，交给 `net/http.Transport` 内部处理；代理不把 chat POST 人为标记成可透明重放。

### 3. 不使用空 `Idempotency-Key` 标记 chat POST

文件：

- `internal/provider/provider.go`

审查时确认：Go `net/http.Transport` 会把带 `Idempotency-Key` 或 `X-Idempotency-Key` 的 POST 视为 replayable。零长度 slice 不会上线发送 header，但仍会改变本地重放语义。对 `chat/completions` 这类非幂等请求，这会扩大标准库内部重放范围，和“请求可能已提交后不重放”的边界冲突。

因此当前做法是：

- 不给 chat POST 设置空 `Idempotency-Key`。
- 保留 provider 层基于 `httptrace` 阶段的手动重试边界。
- `writing_request`、`waiting_response_headers`、`receiving_response_headers` 之后不由代理重放。

踩坑点：

- 空 `Idempotency-Key` 虽然不发到上游，但会让 Go 本地认为 POST 可 replay。
- 对工具调用和计费请求，任何透明重放都必须非常保守；不能为了减少偶发 stale connection 失败而扩大重复提交风险。
- 后续不要重新引入空 `Idempotency-Key`，除非能证明对应请求语义真的幂等，或上游支持可验证的幂等键。

### 4. proxy 层细化错误分类

文件：

- `internal/proxy/diagnostic_error.go`
- `internal/proxy/diagnostic_error_test.go`

新增或强化的分类：

- `upstream_no_response`：请求已经写出，等待或接收响应头阶段失败。
- `upstream_stream_interrupted`：上游 SSE 流已经开始或已建立读取链路，但在合法终态前中断。
- `upstream_api_error`：上游明确拒绝，例如账号、额度、模型、参数等 4xx/业务错误。

诊断摘要中新增 provider 内部尝试明细：

```text
upstream_attempts=N last=<stage>/<category>
```

目的：

- 不再把 `waiting_response_headers` 阶段的 EOF、连接断开笼统写成“无法连接上游”。
- 能看出失败是在 DNS、TCP、TLS、写请求、等待响应头还是流式读取阶段。
- 能辅助判断代理是否发生请求放大。

### 5. 流式读取错误不再误判为协议解析错误

文件：

- `internal/proxy/server.go`
- `internal/proxy/server_test.go`
- `internal/proxy/integration_test.go`

本轮区分两类错误：

- 底层读失败，例如 EOF、unexpected EOF、connection reset、broken pipe：归为 `upstream_stream_interrupted`。
- 终态缺失，例如 OpenAI SSE 没有 `finish_reason/[DONE]`、Ollama NDJSON 没有 `done=true/[DONE]`：也应视为 `upstream_stream_interrupted`，不要把半截流伪装成成功，更不要硬塞成纯 `proxy_parse_error`。
- 应用层协议错误，例如 SSE 过大、JSON 解析失败、工具协议不完整：继续归为 `proxy_parse_error`。

踩坑点：

- 不能把所有 scanner 错误都归为上游流中断，否则真实协议处理 bug 会被掩盖。
- 不能在已经向 VS 写过 token 后 fallback 到另一种模式，否则客户端可能看到两段不一致输出，或者触发重复工具调用。
- 2026-07-23 追加审查发现：最初只补强了 OpenAI SSE 直通路径，`Ollama NDJSON -> OpenAI SSE`、`Ollama NDJSON` 原生直通、`OpenAI SSE -> Ollama NDJSON` 三条转换/直通路径仍会把 `scanner.Err()` 的底层读失败直接向外抛出，导致同类上游断流在不同协议面下分类不一致。已统一包装为 `upstream_stream_interrupted`，但仍只包装底层读错，不把 JSON/工具协议解析错误伪装成网络中断。
- HTTP 流式响应一旦已经写出部分 body，代理无法再把客户端看到的 HTTP 状态从 200 改成 502，这是 HTTP 协议限制；此类场景必须依赖 `stream_state=downstream_started`、attempt 失败日志和 `upstream_stream_interrupted` 分类来定位，不能把“客户端最终看到 200”误当作上游完整成功。

### 6. fallback 边界更保守

文件：

- `internal/provider/provider.go`
- `internal/proxy/*`

结构化传输失败不会触发流式/非流式 fallback，尤其是已经进入 `writing_request`、`waiting_response_headers` 或流式已经输出后。

保留的 fallback 场景仍以明确 5xx 或协议兼容兜底为主，但不能把客户端取消、上游 4xx、结构化已提交请求失败拿来重放。

2026-07-23 追加审查结论：`upstream_no_response` 和 `upstream_stream_interrupted` 也必须停止外层候选 fallback。前者表示请求已经写出并等待/接收响应头失败，后者表示流式响应已建立但未到合法终态；二者都不应自动换候选重放同一 chat 请求。当前路由策略默认只执行首个候选，这条判断是额外保险，防止后续策略调整时重新引入请求放大。

### 7. 诊断脚本支持长期重复测试

文件：

- `tests/useai_large_request_diagnostic.sh`
- `tests/large_request_matrix_diagnostic.sh`
- `tests/README.md`

新增或强化：

- `REPEAT`：每个 provider/model/size 组合重复运行。
- `MODE`：支持 direct-only、proxy-only、direct-proxy 等模式。
- 聚合成功率、错误码、`stream_state`、`network_peer`、p50/p95。
- 每轮写入 JSONL，最终生成 summary JSON。
- 输出 `attempts_summary`、`diagnostic_summary`、`cancel_reason`、`fallback_mode`、`effective_timeout_seconds` 等字段。

踩坑点：

- 单轮测试无法判断间歇性问题。1 次成功或 1 次失败都不足以证明稳定性。
- direct-proxy 模式下必须按模式判断成功率：proxy-only 不要求 direct 成功；direct-proxy 需要 direct 和 proxy 都 2xx，否则会误算。
- 矩阵脚本和单次诊断脚本必须在每轮开始时清空旧 `result.json` / `logs.json`，否则当前轮脚本若提前失败，旧结果会被误读，造成稳定性虚报。
- `.bin/useai-large-diagnostic/` 和 `.bin/large-request-matrix/` 是本地诊断产物，不应提交。

## 真实上游复测记录

以下记录来自本轮实际测试结果，不代表永久稳定性，只代表当次环境和时间窗口。

### UseAI 400KB proxy-only 10 轮

配置：

- provider：UseAI
- upstream/model：`deepseek-v4-flash`
- 请求大小：约 400KB
- 模式：proxy-only
- 重复：10 轮

结果：

- 7/10 成功。
- 3/10 失败。
- 失败均归类为 `upstream_stream_interrupted`。
- `stream_state=upstream_connected`。
- `request_bytes=400000`、`upstream_bytes=400000`、`delta_bytes=0`。

结论：

- 当次失败不是代理请求体异常放大。
- 当次失败不是 DNS/TCP 连接未建立阶段的问题。
- 更符合“上游或中间网关已经接入后，SSE 流中途断开”的现象。

### UseAI/DeepSeek direct-proxy 3 轮对照

结果：

- UseAI proxy：3/3 成功。
- UseAI direct：3 轮中 2 轮出现 curl 层失败。
- 官方 DeepSeek direct：3/3 返回 `402`。
- 官方 DeepSeek proxy：3/3 映射为 `upstream_api_error`，`last=http_402/upstream_api_error`。

结论：

- 官方 DeepSeek 的当次 `402` 属于上游账号/API 层拒绝，不应归为代理网络问题。
- UseAI direct 与 proxy 在短窗口下结果不同，说明真实公网链路和上游状态会影响观测；需要多轮 soak，而不是单轮下结论。

## 不能做或不能承诺的事情

1. 不能承诺完全杜绝 `upstream_stream_interrupted`。
2. 不能承诺完全杜绝 `client_gone`。
3. 不能在客户端已经取消后继续强行请求上游。
4. 不能在请求可能已经提交给上游后由代理手动重放。
5. 不能把所有 EOF 都当成网络问题；应用层解析错误仍必须暴露为 proxy bug 或协议不兼容。
6. 不能把最近窗口成功率当成长期 SLA。
7. 不能把 UseAI/new-api/sub2api/官方 DeepSeek 的真实上游波动说成代理已经彻底根治。

## 仍然可能发生 `client_gone` 的原因

以下是工程判断，不是本轮代码能完全证明的唯一根因：

- VS/Copilot 客户端达到自己的等待上限后主动断开。
- 用户主动取消请求或切换会话。
- 本地网络、代理、VPN 或系统睡眠导致客户端连接断开。
- 上游首 token 太慢，客户端先放弃。
- 大上下文工具请求导致整体延迟上升，触发客户端等待阈值。

应对方向：

- 降低单次上下文大小。
- 选择首 token 更稳定的模型/渠道。
- 客户端侧适当提高等待时间或避免中途取消。
- 通过 `recent_stability` 和日志中的 `cancel_reason` 区分客户端取消与上游中断。

## 2026-07-23 追加：防止收尾阶段把上游失败覆盖成 `client_gone`

发布后现场日志出现一个严重现象：控制台最终请求日志显示 `502 upstream_no_response`，但界面或流状态同时看到大量 `client_gone`。这类不一致不能简单归因为用户取消，因为同一条请求的主错误码、响应状态和 attempts 明细已经指向 `waiting_response_headers/upstream_no_response`。

复查后确认风险点在 `responseWriter.finalizeRequestStatus`：请求 handler 返回后，如果 `r.Context().Err()` 非空，旧逻辑会把 `X-Proxy-Error-Code` 覆盖成 `client_gone`。这在没有任何明确失败归因时可以作为兜底，但一旦 handler 已经写入 `upstream_no_response`、`upstream_stream_interrupted`、`upstream_server_error` 等主因，收尾阶段再覆盖就是误报。

修正原则：

- `client_gone` 只能表示“下游连接生命周期事件”，不能覆盖已经明确的上游或代理主因。
- 如果状态码已经是 4xx/5xx，或响应头已经有 `X-Proxy-Error-Code`，`finalizeRequestStatus` 必须保持原诊断不变。
- `cancel_reason` 只用于细分 499/取消类请求；对于明确的 502 上游失败，应保持为空，避免聚合统计把上游问题算进客户端断开。
- 新增回归测试 `TestLoggingMiddlewareDoesNotOverwriteExplicitUpstreamFailureWhenContextCanceled`，模拟“上游无响应已写入诊断，随后 request context 取消”，要求最终日志仍为 `502 upstream_no_response`。
- 追加优化：`ShouldAttemptAlternateChatMode` 对带 `UpstreamAttempts` 的真实上游 5xx 也不再放行流式/非流式互切。原因是这类错误已经说明 chat POST 到达过上游，继续重放同内容请求只会放大费用和重复工具调用风险；兼容旧行为只保留给没有结构化尝试明细的简化错误。

这个修复不承诺减少真实客户端断开，但能减少本程序把上游失败虚报为 `client_gone` 的概率。后续如果继续看到大量 `client_gone`，必须先按 request_id 对齐最终日志、状态码、`error_code`、`cancel_reason`、`stream_state` 和 attempts，再判断是否是真实客户端侧批量断开。
