# 版本检查与自动下载设计记录

日期：2026-07-04

本文记录 `vs-ai-proxy` 版本检查和自动下载功能的实现设计、使用方式和已验证事实。

## 1. 功能边界

本次实现包含三种更新入口：

- 普通启动：正式版本启动后先自动执行自更新检查；发现新版本时自动下载、安装并重启到新版。
- `--self-update`：命令行主动触发同样的检查、下载、安装和重启流程。
- `--update`：只检查并下载匹配当前平台的 GitHub Release 资产，解压后等待用户自行处理。

启动自动更新默认开启，可通过 `VS_AI_PROXY_AUTO_UPDATE=0`、`false`、`no` 或 `off` 关闭。启动自动更新使用较短超时，失败时只记录警告并继续启动当前版本，避免网络不可用、GitHub 匿名限流或 PowerShell 启动失败导致服务不可用。

历史说明：`v0.2.14` / `v0.2.15` 已具备 `--self-update`，但普通双击/普通启动不会触发更新；Windows 上双击控制台程序也不会显示持久 GUI，因此容易被误判为“打开后没有任何反应”。后续版本修正为普通启动也执行自动自更新。

命令模式包括：

- `--update`：检查新版本并自动下载匹配当前平台的 GitHub Release 资产，解压后等待用户自行处理。
- `--self-update`：检查、下载、校验、备份当前二进制、替换为新版，并自动启动新版进程。

`--update` 是低风险下载模式，不覆盖正在运行的当前可执行文件，也不会自动重启服务。

`--self-update` 是全自动安装模式，行为如下：

1. 查询 GitHub Latest Release。
2. 比较当前版本和最新版本。
3. 选择当前 `GOOS/GOARCH` 对应的 release asset。
4. 下载压缩包到更新目录。
5. 计算 SHA256。
6. 如 release 中存在 checksum 文件，则校验 checksum。
7. 解压出 `vs-ai-proxy` 或 `vs-ai-proxy.exe` 到更新目录。
8. 将新版二进制复制到当前可执行文件旁边的 `.new` 文件。
9. 将当前可执行文件备份为 `.bak-<timestamp>`。
10. 用 `.new` 替换当前可执行文件；如果替换失败，尝试把备份恢复回原路径。
11. 在 macOS/Linux 上使用 `exec` 原地切换到新版进程，避免旧进程尚未释放端口时新版抢占失败。
12. 从重启参数中移除 `--self-update`、`--update`、`--check-update`、`--version` 和 `--update-dir`，避免重复自更新或新旧版本命令参数不兼容。
13. Windows 上由于运行中的 `.exe` 通常不能直接覆盖，会启动 PowerShell 延迟替换脚本，等待当前进程退出后完成替换并重启。

## 2. 命令

新增命令行参数：

```bash
vs-ai-proxy --version
vs-ai-proxy --check-update
vs-ai-proxy --update
vs-ai-proxy --self-update
vs-ai-proxy --update --update-dir /path/to/updates
vs-ai-proxy --self-update --update-dir /path/to/updates
```

默认更新目录：

```text
<config-dir>/updates
```

其中 `<config-dir>` 使用现有配置目录规则，即默认 `~/.config/vs-ai-proxy`，或受 `XDG_CONFIG_HOME` 影响。

## 3. GitHub API 与鉴权

默认查询：

```text
https://api.github.com/repos/qwdingyu/vs-ai-proxy/releases/latest
```

实现位置：

- `internal/update/update.go`

请求头：

- `Accept: application/vnd.github+json`
- `User-Agent: vs-ai-proxy-updater`
- 如果环境变量 `GITHUB_TOKEN` 存在，则附加 `Authorization: Bearer <token>`。

支持 `GITHUB_TOKEN` 的原因：匿名 GitHub API 请求可能遇到 rate limit。实现过程中实际遇到过匿名 403 rate limit，加入 token 支持后真实检查和下载通过。

### 3.1 网络异常时的手动更新提示

Windows exe 控制台必须在更新检查、下载或自更新失败时显示可复制的手动更新入口：

```text
GitHub 最新版本: https://github.com/qwdingyu/vs-ai-proxy/releases/latest
```

设计原因：

- 用户网络可能无法直接访问 GitHub API 或 Release CDN，但可以把链接复制到其它网络环境、浏览器、代理工具或同事机器上下载。
- 当程序已经成功拿到 Release 元数据、但下载资产失败时，控制台还应额外显示当前平台资产 URL 和 Release URL，方便直接下载对应 Windows x64 包。
- 当程序连最新版本元数据都无法获取时，仍然显示稳定的 `releases/latest` 页面；该页面由 GitHub 跳转到最新 Release，不依赖客户端知道具体版本号。

这不是新的更新源策略，也不是绕过校验逻辑。自动更新仍按 `Check -> Download -> checksum -> extract -> self-update` 主链路执行；手动链接只是失败后的可恢复操作指引。

## 4. 平台资产匹配规则

根据当前 `GOOS/GOARCH` 选择资产：

| GOOS/GOARCH | Release asset 后缀 |
| --- | --- |
| `darwin/amd64` | `macos-x64.tar.gz` |
| `darwin/arm64` | `macos-arm64.tar.gz` |
| `linux/amd64` | `linux-x64.tar.gz` |
| `linux/arm64` | `linux-arm64.tar.gz` |
| `windows/amd64` | `windows-x64.exe.zip` |

资产名前缀格式：

```text
vs-ai-proxy-v<version>-<platform>
```

例如：

```text
vs-ai-proxy-v0.2.13-macos-x64.tar.gz
```

## 5. 版本比较

当前版本由构建时 ldflags 注入：

```text
-X main.version=<version>
```

正式 release 版本按 `vX.Y.Z` 或 `X.Y.Z` 比较。

如果当前版本是 `dev` 或空值，则 `--check-update` / `--update` 返回错误，避免把开发构建与正式 release 做不可靠比较。

本地单元测试：

```bash
go test ./internal/update ./cmd/server -count=1
go test ./...
```

真实 GitHub Release 检查和下载验证：

```bash
tmpbin="$(mktemp -d)/vs-ai-proxy"
go build -ldflags='-s -w -X main.version=0.2.12' -o "$tmpbin" ./cmd/server
token="$(gh auth token)"
GITHUB_TOKEN="$token" "$tmpbin" --check-update
tmpupdates="$(mktemp -d)"
GITHUB_TOKEN="$token" "$tmpbin" --update --update-dir "$tmpupdates"
```

验证结果：

- `--check-update` 发现 `0.2.12 -> v0.2.13`。
- 当前机器匹配资产为 `vs-ai-proxy-v0.2.13-macos-x64.tar.gz`。
- `--update` 成功下载 release 资产。
- `--update` 成功计算 SHA256。
- `--update` 成功解压出 `vs-ai-proxy` 二进制。
- 没有替换当前正在运行的程序。

## 8. 后续可选增强

以下能力本次未实现，原因是需要更明确的安装/服务管理策略：

- 原生 systemd/launchd/Windows Service 管理器集成。
- 增量更新。

## 9. 2026-07-12 追加：Windows 后台自更新脚本加固

### 9.1 背景

Windows 下正在运行的 `vs-ai-proxy.exe` 不能被当前进程直接覆盖，因此自动更新采用：

1. 下载并解压新版二进制。
2. 写入 `vs-ai-proxy.exe.new`。
3. 生成 `vs-ai-proxy-self-update.ps1`。
4. 当前进程退出。
5. PowerShell 脚本等待旧进程退出后完成 `.new -> exe` 替换并重启。

真实用户环境中，如果杀毒软件、文件锁、权限或进程退出延迟影响替换，可能残留：

- `vs-ai-proxy.exe.new`
- `vs-ai-proxy-self-update.ps1`
- 旧 exe 仍未替换

这会让用户误以为“已经更新成功”，但实际仍运行旧版本。

### 9.2 加固内容

后台脚本现在包含：

- **预检**：确认 `.new` 存在且非空。
- **重试替换**：备份旧版本和安装新版本都最多重试 20 次，每次间隔 300ms。
- **明确日志**：每一步写入 `vs-ai-proxy-self-update.log`。
- **捕获红字**：PowerShell 子进程的 stdout/stderr 也追加到同一个日志，控制台一闪而过的红色错误不能只停留在屏幕上。
- **启动阶段记录**：Go launcher 在启动 PowerShell 前后记录 `launcher prepared`、`launcher started powershell pid=...`；如果 PowerShell 启动失败，记录 `launcher failed to start powershell: ...`。
- **残留检测**：安装后如果 `.new` 仍存在，直接报错，避免假成功。
- **回滚**：如果 exe 缺失且备份存在，尝试恢复备份。
- **参数保留**：重启时保留原本的非更新参数，例如 `--config`。

### 9.3 排障顺序

如果用户反馈“显示已安装新版本，但 exe 还是旧版本”：

1. 查看同目录 `vs-ai-proxy-self-update.log`。
2. 如果日志没有 `launcher prepared`，说明还没有进入 Windows 延迟替换阶段，优先查下载、校验、解压或 `.new` 暂存步骤。
3. 如果有 `launcher failed to start powershell`，说明 PowerShell 启动阶段失败，优先查执行策略、系统权限、杀毒软件或 PowerShell 是否可用。
4. 如果日志停在 `waiting for pid`，说明旧进程未退出或被其他方式拉起。
5. 如果日志有 `备份旧版本失败` 或 `安装新版本失败`，优先查文件锁、杀毒软件、权限。
6. 如果日志有 `ERROR_RECORD`、`ERROR_STACK` 或 `ERROR_POSITION`，这些就是之前控制台红字一闪而过的持久化错误详情。
7. 如果 `.new` 仍存在，说明替换没有完成，不应认为更新成功。
8. 如果 `rollback restored backup` 出现，说明脚本已恢复旧版本，需根据前面的错误处理根因。

### 9.4 仍不做的事情

- 不强杀任意非本程序进程。
- 不绕过系统权限策略。
- 不把更新失败伪装为成功。
- 不引入 Windows Service 管理器；这是后续独立安装形态问题。

### 9.5 更新缓存文件的鲁棒替换

更新目录中可能残留旧的 release 压缩包、解压后的 `vs-ai-proxy` / `vs-ai-proxy.exe`，尤其是用户反复执行 `--update` 或上一次自动更新中断后。

缓存层替换策略是：

1. 先把新内容完整写入 `.tmp`。
2. 优先尝试 `.tmp -> 目标文件` 的原子重命名。
3. 如果平台不允许覆盖已有目标文件，例如 Windows 普通文件已存在，则把旧目标移动到唯一 `.bak-<timestamp>`。
4. 再把 `.tmp` 移动为目标文件。
5. 如果第 4 步失败，立即把 `.bak-<timestamp>` 回滚回目标路径。
6. 成功后清理本次唯一备份；如果清理失败，不影响已安装的新缓存文件。

边界说明：

- 这个策略只用于更新缓存目录里的压缩包和解压后二进制，不用于正在运行的正式 exe。
- 正式 exe 的替换仍由 `replaceExecutable` 或 Windows 延迟 PowerShell 脚本处理。
- 不直接删除旧目标文件再重命名，因为这样会在替换失败时丢失用户已有缓存，造成后续排障更困难。
- 如果目标路径是目录，直接报错，不尝试覆盖目录。
