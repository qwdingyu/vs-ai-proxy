# 文档索引与系统架构总览（2026-07-14）

本文是 `docs/` 目录的入口文档，用于避免历史踩坑记录分散后难以维护。详细问题仍以各专题文档为准；本文只记录系统全貌、运行机制、扩展边界和排障入口。

## 1. 产品定位

`vs-ai-proxy` 起步于 **Windows + Visual Studio Copilot BYOM** 场景，是本地 OpenAI-compatible 代理。随着官方 Copilot 自带模型 / 自定义 endpoint 能力推进，长期定位不能只停留在“让 Visual Studio 接上第三方模型”，而应演进为面向 AI 编程客户端的本地兼容、诊断、路由与成本治理层。产品定位和仓库策略的完整决策记录见 `32_产品定位与仓库策略_20260718.md`。

核心原则：

1. **配置简单**：小白用户只需要配置 provider 的 `Base URL` 和 `API Key`。
2. **协议保真**：OpenAI `tool_calls`、legacy `function_call`、流式工具分片默认尽量透传。
3. **最小兼容**：只修正 VS/Copilot 明确不兼容的协议细节，如空 `finish_reason`、`stream=false` 返回 SSE、DSML 方言。
4. **可诊断**：所有失败应尽量落到可解释的 `error_code`、`reason`、`action`、`diagnostic_summary`。
5. **不越权**：不伪造工具调用，不在下游流已开始后自动切 provider，不把文件/搜索工具降级成 shell 工具。

## 3. 运行机制

### 3.1 启动流程

1. 解析命令行参数，例如 `--version`、`--check-update`、`--self-update`。
2. 加载 `.env` 与运行配置。
3. 加载 `config.json` 中的 providers、models、防御参数和日志配置。
4. 初始化 provider registry、store、Web 管理 API、代理 API。
5. 可选执行启动自动更新检查。
6. 监听本机端口，默认 `127.0.0.1:12345`。

### 3.2 请求流程

```text
客户端请求 /v1/chat/completions
  -> 解析 OpenAI ChatRequest
  -> 记录 request_tools / 请求体大小
  -> 根据 model 解析 provider/model 候选
  -> 参数治理（不兼容参数、timeout、防御模式）
  -> 调用上游 provider
  -> 按响应模式处理：
       - 非流式 JSON：最小归一化后返回
       - 流式 SSE：标准事件解析 + 错误/终态校验 + finish_reason/tool_calls 修正
       - stream=false 但返回 SSE：复用统一 collector 聚合为 JSON，保留工具调用
       - DSML 文本方言：在声明工具时转换为标准 tool_calls
  -> 记录 response_tools / fallback_mode / stream_state
  -> 从最终采用响应提取上游权威 usage；缺失保持 unknown，不做本地估算
  -> 写入请求日志和控制台日志
```

Provider 参数治理必须以 `internal/provider/capabilities.go` 为事实源。路径差异、`reasoning_effort` / `top_k` 支持情况，以及 MiMo 这类 `max_completion_tokens` 输出预算字段方言，都应收敛到 provider capability；不要在 `/v1/chat/completions` handler 中按模型名散落特判。新增 provider 或特殊模型前，先按 `34_大模型提供商与特殊参数接入验证指南_20260718.md` 的准入流程验证；MiMo 具体踩坑记录见 `33_MiMo_provider参数方言与VS_Copilot兼容踩坑记录_20260718.md`。

流式链路以 `[DONE]` 作为应用层终态，不等待上游关闭 transport body；工具尾部、finish 和 `[DONE]` 必须在完整性校验后提交。OpenAI refusal-only 响应属于合法结果，raw、typed、SSE 和 Ollama 转换都不得把它误判为空响应。

Token 统计保持“一次客户端请求一条日志”：OpenAI 非流 JSON、SSE 最终 usage、Ollama 原生终态计数和模式 fallback 都归一到可空 `RequestLog.Usage`。`nil` 表示上游未上报，非空全零表示明确上报零值；cached/reasoning 分别是输入/输出子集，不重复加到 total。统计同时维护累计、按模型、按日、按 ISO 周、按月口径，周期请求数包含 unknown usage 请求以暴露覆盖率。累计和周期统计保存在 `logs.stats.json`，`logs.json` 继续保持旧版数组格式以支持 EXE 回滚。字段说明见 `14_Token用量字段说明与统计口径_20260717.md`；详细设计与 API-Switch 源码核验见 `30_API-Switch模型Token统计分析与实施方案_20260717.md`。

### 3.3 工具调用机制

工具调用兼容覆盖以下协议形态：

| 协议形态 | 处理方式 |
| --- | --- |
| OpenAI `tool_calls` | 默认透传，按当前请求声明做安全别名归一化 |
| legacy `function_call` | 默认保留并归一化工具名 |
| 流式 `tool_calls` 分片 | 合并参数分片，保留 `finish_reason=tool_calls` |
| DSML 文本方言 | 仅在请求声明了对应工具/安全别名时转换 |
| `stream=false` 返回 SSE | 聚合成非流式 JSON，并保留 `tool_calls` |
| 流式/非流式同 provider fallback | 同时保留 modern `tool_calls` 与 legacy `function_call` |

标准 SSE 解析只在 `internal/provider.CollectOpenAIChatSSE` 维护；proxy 层只负责 VS finish reason、工具精确声明名、别名和 DSML，不得再复制一套 SSE 聚合器。完整工具调用必须使用工具结束原因；`length/content_filter` 不得暴露可执行工具。

通用工具支持以协议不变量为准，不以工具名目录为准：任意请求已声明工具都必须保留 schema、history、tool choice 和扩展字段；modern 响应提交前必须具备稳定 `id`、`type=function`、完整 name 和合法 JSON arguments。详细矩阵见 `25_VisualStudio_Copilot通用工具调用契约与矩阵测试_20260714.md`。

重要边界：

- 不默认阻断未声明标准工具调用，避免破坏 Copilot stable 行为。
- 不将文件工具映射到 shell 工具。
- 不伪造模型没有返回的工具调用。
- `length/content_filter` 对标准工具和 DSML 方言使用同一条“不执行截断工具”规则。
- 下游 SSE 已开始写出后，不再自动切 provider；后续协议错误仍记录为 provider 失败。

## 5. 扩展指南

### 5.1 新增 provider

1. 优先确认上游是否 OpenAI-compatible。
2. 在 `internal/provider` 中复用 OpenAI provider 能力，避免新增协议层。
3. 如果必须新增协议，补齐：非流式、流式、工具调用、错误分类、测试页测试。
4. 增加 `/models` 或模型导入行为测试。

### 5.2 新增工具别名

1. 只添加语义同族别名。
2. 只能映射到当前请求已声明的目标工具。
3. 文件/搜索工具不要映射到 shell。
4. 必须补 `tool_aliases`、流式 E2E 和 DSML 相关测试。

### 5.3 新增诊断错误类型

1. 先确认错误发生阶段：配置、路由、上游、网络、客户端断开、协议解析。
2. 在响应头、请求日志、控制台日志中保持一致。
3. 提供 `reason/action/summary`，避免只写技术错误正文。
4. 补管理页或日志页可读性验证。

## 8. 已知边界

- 代理无法强制 VS/Copilot 自动压缩上下文，只能通过日志提示和模型选择降低风险。
- 上游 new-api/sub2api 内部渠道慢首 token、429、503、context canceled 不能由本地代理完全消除。
- `upstream_connecting` 只证明尚未收到上游 HTTP 响应头/可读流；新版本应结合错误中的 `upstream_stage` 区分 DNS、连接、TLS、写请求和等待响应头。`waiting_response_headers` 仍不能继续区分网关内部排队和模型推理。
- 下游流已经开始后，不能安全切换 provider。
- 自动更新涉及 Windows 文件锁和安全软件，必须依赖后台脚本日志定位。
- Token 统计只汇总上游明确返回的 usage，不是 provider 账单；没有 usage 的请求显示 unknown，并通过上报覆盖率暴露数据完整性。
- Visual Studio 是当前第一入口，但不是长期唯一身份；任何 Pro、拆仓、授权或自动路由工作，都必须证明其在官方第三方模型能力开放后仍有价值。

## 10. 维护清单

### 10.1 修改工具调用相关代码前

- 确认改动属于哪条协议路径：非流式 JSON、流式 SSE、legacy `function_call`、DSML、`stream=false` 返回 SSE。
- 先补失败用例，再改实现。
- 能稳定复现核心问题的测试必须保存为仓库正式测试，禁止仅用临时 overlay 验证后删除。
- 必须运行 `make tool-check`。
- 禁止为了修一个工具，把不同语义工具互相映射，例如把 `create_file` 映射到 `powershell`。

### 10.2 修改模型路由/provider 前

- 检查 provider_id、模型展示名、上游模型名是否一致。
- 同一模型短名匹配多个 provider 时，不能静默随机路由。
- 测试页模型下拉必须来自所选 provider 的实际 `/models` 或导入结果。
- 新 provider 或特殊参数必须先按 `34_大模型提供商与特殊参数接入验证指南_20260718.md` 判断归属层，禁止直接在 handler 中追加 provider/model 特判。

### 10.3 修改日志/诊断前

- 控制台日志、Web 日志、响应头错误码要保持同一语义。
- 不要只写“上游错误”，必须尽量区分 400/401/403/413/429/499/5xx/network/timeout。
- 对工具调用失败，优先同时看 `request_tools` 和 `response_tools`。
- 对 499，必须结合 `elapsed_ms`、`stream_state`、请求体大小判断，不要一律写成用户手动取消。

### 10.4 修改 Web 国际化代码前

- `data-i18n` 只能挂在叶子文本节点，表单控件必须保留为兄弟节点。
- 新增 HTML 或 `t()` 键后，必须同时更新中英文词典；禁止重复定义键。
- 修改 `/admin` 静态资源时，必须验证 `/admin/i18n/*` 返回 JavaScript，而不是 SPA `index.html`。
- 至少运行 `make i18n-check` 和 `go test ./web ./internal/api -count=1`。

### 10.5 修改自动更新/Release 前

- 必须确认 Windows x64 zip 包存在且可下载。
- Windows 自更新失败时，应能看到 `.new`、`.ps1`、`.log` 位置。
- GitHub Release 刚发布时可能存在资产/CDN 短暂不可见，更新逻辑必须容忍短时重试。

### 10.6 发版前最低验证

```bash
make tool-check
go test ./internal/proxy ./internal/provider -count=1
make release-check
```

如果修改了 Web 管理页，还应手工检查：提供商、测试、日志、FAQ、联系我们、高级菜单是否可打开，日志页是否能展示诊断字段。
