# Copilot 上下文压力治理方案（2026-07-12）

## 1. 背景

近期真实使用中反复出现以下现象：

- 旧 VS Copilot session 中同一模型持续失败，常见表现为 `499 client_gone`、`client_deadline_reached`、`502 network_error`、`context canceled`。
- 重新新建 session 后，同一模型、同一 provider 又能恢复正常。
- 请求日志中经常可以看到较大的 `request_bytes` / `upstream_bytes`，例如数百 KB 到 1MB 以上。

这说明问题不能只按“模型不可用”或“网络故障”解释。更稳妥的判断是：旧 session 中历史消息、文件片段、工具调用记录和工具结果不断累积，导致请求体膨胀；请求体越大，上游首 token 越慢，网关/连接越容易超时或被客户端等待上限打断。

本方案目标不是让代理替代 Copilot 做复杂记忆管理，而是在不破坏工具调用主链路的前提下，给出低风险、可观测、可回滚的治理策略。

## 2. 核心结论

### 2.1 不能依赖 Copilot 自动压缩上下文

VS Copilot 客户端是否压缩、如何压缩上下文，不在 `vs-ai-proxy` 的直接控制范围内。代理只能看到最终发来的 HTTP 请求，不能可靠控制 Copilot 客户端如何选择文件、历史消息和工具状态。

因此，代理侧不能把“让 Copilot 自动压缩上下文”作为可交付能力。代理能做的是：

1. 识别上下文压力。
2. 在日志和诊断里明确提示。
3. 必要时按配置做保护性拒绝。
4. 在显式开启实验功能时做极保守裁剪。

### 2.2 最轻量级观测和提示是否足够

短期看，**只做轻量级观测和提示基本是最稳妥的默认方案**，原因是：

- 不改写 Copilot 请求，不会破坏工具调用、文件上下文、编辑状态。
- 能让客服从日志中直接看出“旧 session 膨胀/请求体过大/首 token 慢”的排查方向。
- 对现有用户行为零侵入，风险最低。
- 对 new-api/sub2api、多 provider、多模型都通用。

但它不是完整解决方案。只做观测和提示时，代理不会阻止超大请求继续发往上游，所以仍可能出现：

- 用户继续等待到 Copilot 客户端超时。
- 上游返回 413/499/502/503。
- 客服仍需要建议用户新建 session 或减少上下文。

因此建议分阶段推进：

- 默认阶段：观测 + 提示。
- 可选阶段：保护性拒绝。
- 实验阶段：保守裁剪。
- 不建议近期做：自动摘要压缩。

## 3. 推荐分层策略

### L0：观测层，默认开启

记录并展示：

- `request_bytes`
- `upstream_bytes`
- `elapsed_ms`
- `error_code`
- `error_reason`
- `error_action`
- `diagnostic_summary`
- `attempts_summary`
- `stream_state`
- `network_peer`
- `request_tools`
- `response_tools`

目标：先把问题看清楚，不改变请求行为。

### L1：软提示层，默认开启

当请求体达到大上下文阈值时，在日志中追加类似提示：

```text
本次请求体/上游体约 619.2 KB，属于大上下文；如果新建 session 后恢复，优先怀疑旧 session 历史膨胀、文件堆积或状态污染。
```

适用错误：

- `client_deadline_reached`
- `client_gone`
- `timeout`
- `network_error`
- `upstream_payload_too_large`
- `upstream_server_error`
- `proxy_parse_error`

不应追加到以下错误，避免误导：

- `upstream_auth_error`
- `upstream_request_error`
- `upstream_rate_limit`

### L2：保护性拒绝，默认关闭

当请求体超过硬阈值时，代理可以不访问上游，直接返回结构化错误，例如：

```json
{
  "error": {
    "code": "context_pressure_too_large",
    "message": "当前 Copilot session 上下文过大，请新建 session 或减少打开文件后重试。",
    "details": {
      "request_bytes": 1800000,
      "limit_bytes": 1572864
    }
  }
}
```

优点：

- 避免用户白等 60-100 秒。
- 避免把明显过大的请求继续打到上游。
- 客服口径明确。

风险：

- 用户可能认为“代理拒绝了正常请求”。
- 不同模型实际上下文能力不同，硬阈值需要可配置。

因此必须默认关闭，只允许用户显式开启。

### L3：保守裁剪，实验功能，默认关闭

如果后续要做裁剪，只能裁剪最老的普通历史消息，必须保留：

- `system` 消息。
- 当前最后一条 `user` 消息。
- 所有 `tools` 声明。
- 最近 N 轮 `assistant tool_calls`。
- 与最近工具调用对应的 `tool` 结果。
- 最近包含文件路径、错误堆栈、diff、编辑结果的消息。

必须记录：

- 裁剪前体积。
- 裁剪后体积。
- 裁剪消息数量。
- 是否保留工具链。
- 是否触发裁剪。

响应头和日志应明确：

```text
context_trimmed=true
context_trim_before_bytes=...
context_trim_after_bytes=...
context_trim_removed_messages=...
```

### L4：自动摘要压缩，不建议近期实现

自动摘要压缩风险最高：

- 需要额外模型调用，增加延迟和成本。
- 摘要可能丢失工具调用状态。
- 摘要可能误改用户意图。
- 对文件编辑类任务，摘要错误会直接导致错误修改。

因此不建议作为近期功能，更不应默认开启。

## 4. 阈值应该按模型，还是按 Copilot

### 4.1 这是两个不同维度

上下文压力有两个限制来源：

1. **Copilot 客户端等待窗口和交互体验限制**。
2. **上游模型/provider 的上下文、网关和首 token 能力限制**。

这两者不是同一个概念。

### 4.2 Copilot 维度：更像硬体验指标

VS/Copilot 常见等待窗口约在 100 秒量级。即使某个模型理论上下文很大，如果旧 session 请求体导致首 token 超过客户端等待上限，用户仍然会看到失败。

因此，针对 Copilot 场景，应该有一个代理侧统一的体验阈值，用来判断“这次请求是否已经进入高风险区”。

建议默认诊断阈值：

| 阈值 | 建议含义 | 默认动作 |
| --- | --- | --- |
| `>= 512KB` | 大上下文 | 日志提示 |
| `>= 1MB` | 超大上下文 | 强提示，建议新建 session |
| `>= 1.5MB` | 极高风险 | 可选保护性拒绝，默认关闭 |

这些阈值不是模型上下文硬上限，而是 Copilot 使用体验和链路稳定性的风险阈值。

### 4.3 模型/provider 维度：应该允许覆盖

不同模型和 provider 的能力确实不同：

- 有的模型上下文更大。
- 有的上游网关 body 限制更小。
- 有的 provider 首 token 更慢。
- 有的 new-api/sub2api 渠道组内部有多渠道轮换。

因此，模型/provider 应该支持覆盖阈值，例如：

```json
{
  "models": [
    {
      "name": "gpt-5.5",
      "provider": "useai2",
      "context_pressure_warn_bytes": 524288,
      "context_pressure_hard_bytes": 1572864
    },
    {
      "name": "deepseek-v4-flash",
      "provider": "deepseek",
      "context_pressure_warn_bytes": 1048576,
      "context_pressure_hard_bytes": 3145728
    }
  ]
}
```

但默认不要让用户必须理解这些细节。默认应按 Copilot 体验阈值工作，只有高级用户或特定 provider 才覆盖。

## 5. 推荐配置模型

建议未来配置形态：

```json
{
  "defense": {
    "context_pressure_enabled": true,
    "large_context_warn_bytes": 524288,
    "large_context_strong_warn_bytes": 1048576,
    "large_context_hard_bytes": 1572864,
    "context_reject_enabled": false,
    "context_trim_enabled": false,
    "context_trim_target_bytes": 786432,
    "context_trim_preserve_recent_messages": 8
  }
}
```

默认：

- `context_pressure_enabled = true`
- `context_reject_enabled = false`
- `context_trim_enabled = false`

也就是说，默认只观测和提示，不拦截、不裁剪、不摘要。

## 6. 产品建议

### 当前阶段建议

当前阶段只做 L0/L1，即：

- 继续记录请求体和上游体。
- 在日志中输出大上下文提示。
- 在错误诊断中告诉用户：如果新 session 后恢复，优先怀疑旧 session 上下文膨胀。
- Web 日志页支持按 `error_reason`、`error_code`、provider、model、request_id 筛选。

这已经能显著减少客服沟通成本。

### 下一阶段建议

如果用户仍频繁遇到旧 session 大请求导致的超时，可以新增 L2：

- Web 端增加开关：`超过硬阈值时保护性拒绝`。
- 默认关闭。
- 开启后返回结构化错误，不访问上游。
- 日志记录拒绝原因和体积。

### 不建议近期做

不建议近期实现默认自动裁剪或摘要压缩。原因是它们会改变 Copilot 发来的语义上下文，容易破坏工具调用和文件编辑链路。

## 7. 测试准入要求

如果后续实现 L2/L3，至少需要以下测试：

1. 小请求完全不变。
2. 大请求默认只提示，不改写。
3. 开启保护性拒绝后，不访问上游。
4. 拒绝响应包含结构化错误和体积信息。
5. 开启裁剪后 JSON 仍合法。
6. 裁剪后保留 `system`。
7. 裁剪后保留最后 `user`。
8. 裁剪后保留 `tools`。
9. 裁剪后保留最近 `tool_calls` 与对应 `tool` 结果。
10. 裁剪动作写入日志和响应头。
11. `make tool-check` 必须通过。
12. 流式和非流式都必须覆盖。

## 8. 最终建议

现阶段最优方案是：

```text
默认：观测 + 提示
可选：保护性拒绝
实验：保守裁剪
暂缓：自动摘要压缩
```

这能最大限度降低风险，同时把旧 session 上下文膨胀导致的问题从“客服猜测”变成“日志可解释”。

## 9. 控制台日志与 Web 日志展示要求

上下文压力治理必须同时体现在两类日志中：

### 9.1 控制台日志

控制台请求完成日志除 `request_id` 和 `error_code` 外，还应输出：

- `reason`：一句话原因，例如 `网络/CDN/连接异常`。
- `action`：下一步处理建议，例如检查上游渠道健康或新建 session。
- `summary`：压缩摘要，包含耗时、请求体、上游体、远端 peer、流状态和上下文压力提示。
- `attempts`：候选 provider/model 的耗时与错误分类。
- `request_bytes` / `upstream_bytes`：人类可读体积。

这样即使用户不打开 Web 管理页，维护人员也能从控制台直接判断是否属于旧 session 上下文膨胀、上游 5xx、网络/CDN、413 或客户端等待上限。

### 9.2 Web 日志

Web 日志页必须展示同一套字段：

- 表格错误徽标显示 `error_code · error_reason`。
- tooltip 展示 `error_action`、`diagnostic_summary`、`attempts_summary`、请求体、上游体、超时、远端 peer、流状态。
- 最近失败卡片展示全局 latest failure，而不是只看当前分页。
- 支持按 `error_reason`、`error_code`、provider、model、request_id、status_code 筛选。

控制台日志和 Web 日志必须使用同一套后端诊断函数，避免两边口径不一致。
