# Web 管理页国际化资源路由与 DOM 安全踩坑记录（2026-07-14）

## 1. 背景

`web/dist/index.html` 新增中文/英文切换后，静态检查和 Go 字符串测试均能通过，但严格审查发现国际化功能没有形成从浏览器资源加载、DOM 渲染、动态文案到发布门禁的完整闭环。

本次修复只围绕已有 Web 管理页和单端口 EXE 服务展开，不引入前端框架或运行时依赖。目标是：

1. `/admin` 页面能够从嵌入式静态文件加载全部 i18n 脚本。
2. 翻译初始化不会删除输入框、选择框、文本域或按钮。
3. 中英文词典键、占位符和动态文案保持一致。
4. 发布检查能够在没有真实上游的情况下稳定发现回归。

## 2. 现象

### 2.1 资源加载失败

页面原来使用相对路径：

```html
<script src="i18n/index.js"></script>
```

产品入口是 `http://127.0.0.1:<port>/admin`。浏览器会把该路径解析为 `/i18n/index.js`，而统一服务只把 `/admin` 和 `/admin/*` 分给管理端；请求实际返回 404。

即使用户手工访问 `/admin/`，`/admin/i18n/index.js` 也会落入 SPA fallback，返回 301/`index.html`，不会返回 JavaScript。

### 2.2 翻译初始化删除控件

运行时原来对所有 `[data-i18n]` 执行 `el.textContent = dict[key]`。部分翻译标记直接放在包含表单控件的 `<label>` 上，例如：

```html
<label data-i18n="config.form.port">
  代理端口 <input id="cfgPort" />
</label>
```

`textContent` 会删除 label 的全部子节点，导致配置、provider、模型和测试页面的控件从 DOM 消失。这个问题只有在真实 `DOMContentLoaded` 后才会出现，原有 Go 测试没有覆盖。

### 2.3 词典和动态文本不完整

- `monitor.table.cooldown` 重复定义，后一个带 `{0}` 的值覆盖表头值。
- `faq.vsGuide.screenshot` 被引用但中英文词典都缺失。
- 动态日志按钮、测试按钮 loading 文案和错误 fallback 仍硬编码中文。
- 英文输出混入中文标点；中文配置摘要反而使用英文标签。
- 页面固定 `lang="zh-CN"`，切换英文后屏幕阅读器语言语义错误。

## 3. 排查过程

1. 运行 `go test ./web -count=10` 和 `go test ./...`，确认原有门禁全绿。
2. 启动本地服务并请求四个路径：

   | 路径 | 修复前结果 | 说明 |
   | --- | --- | --- |
   | `/admin` | 200 `text/html` | 页面本身可返回 |
   | `/i18n/index.js` | 404 | 相对 URL 的实际解析结果 |
   | `/admin/i18n/index.js` | 301 | 被 SPA fallback 接管 |
   | `/admin/assets/images/...` | 200 | 现有静态资源路由可复用 |

3. 将资源响应、DOM 标记、词典完整性和动态文案分别固化为失败测试。
4. 先修 `/admin/i18n/*` 静态路由和绝对 URL，再修翻译节点结构；最后处理词典和动态文本，避免用多个互相掩盖的补丁。

## 4. 根因

### 4.1 静态资源边界缺少明确契约

管理页使用 `/admin` 前缀，但后端只显式处理了 `/admin/assets/*`，其他 `/admin/*` 都按 SPA 页面处理。新增资源没有同步更新服务器路由和冒烟测试。

### 4.2 翻译 API 的节点契约不清晰

`data-i18n` 同时被当成“元素全文替换”和“标签文字翻译”使用，没有规定只能放在叶子文本节点，导致父节点替换破坏子控件。

### 4.3 词典缺少机器可验证的完整性约束

原有测试只检查部分 HTML 字符串，不检查：

- HTML 和动态 `t()` 引用是否都存在于两套词典；
- 词典是否重复定义键；
- 中英文占位符是否一致；
- 英文值是否混入中文；
- 运行时是否真的执行了属性翻译和语言切换。

## 5. 解决方案

### 5.1 统一资源 URL 和后端路由

- 页面改用 `/admin/i18n/index.js`、`/admin/i18n/zh.js`、`/admin/i18n/en.js`。
- 管理端静态中间件在 SPA fallback 之前显式服务 `/admin/i18n/*`。
- `internal/api` 增加 JavaScript 状态码、内容类型和响应体测试。
- `build-test.sh` 增加已登录/未登录场景的 i18n 资源冒烟检查。

选择绝对 `/admin/i18n/*` 是因为 EXE 服务的管理入口固定为 `/admin`；不依赖尾部斜杠，也不依赖浏览器对相对 URL 的特殊处理。

### 5.2 统一 DOM 翻译契约

- 所有表单 label 改为 `<label><span data-i18n="...">文字</span><input ... /></label>`。
- 复选框的控件和文字也分别保留为兄弟节点。
- 运行时遇到带子元素的 `data-i18n` 节点直接跳过，不再破坏 DOM。
- `data-i18n-placeholder`、`data-i18n-aria-label`、`data-i18n-alt`、`data-i18n-title` 继续只更新对应属性。
- 初始化和切换时同步 `document.documentElement.lang`：中文为 `zh-CN`，英文为 `en`。

叶子节点规则是边界约束，不是依赖开发者记忆的约定；静态测试会阻止新的 label 父节点误挂翻译标记。

### 5.3 统一词典和动态文案

- 静态表头 `monitor.table.cooldown` 与动态 `monitor.table.cooldownSeconds` 分离。
- 三张 Visual Studio 截图使用独立的 `screenshot1/2/3` alt 键。
- 配置摘要、模型元数据能力、provider 健康诊断使用完整句级翻译，调用方不再拼接中文标点。
- 日志操作按钮、测试 loading、错误 fallback 全部通过 `t()` 获取。

## 6. 测试与门禁

### 6.1 Go 测试

`web/index_html_test.go` 现在检查：

- i18n 脚本使用 `/admin/i18n/*`；
- `data-i18n` 不直接挂在 label 父节点；
- 中英文所有引用键均存在；
- 词典无重复键；
- 中英文占位符集合一致；
- 英文词典不含中文正文；
- 动态用户文案使用 i18n 键。

`internal/api/api_test.go` 检查三个 i18n 资源在管理路由下返回 200、JavaScript Content-Type 和正确文件内容。

### 6.2 零依赖运行时测试

`tests/i18n_runtime_test.js` 直接在 Node VM 中执行正式运行时和词典，覆盖：

- 中文/英文文本查找与 `{0}` 插值；
- placeholder、aria-label、alt、title 属性；
- 带子元素节点不会写入 `textContent`；
- `html lang` 和 `switchLang()` 行为。

通过 `make i18n-check` 执行，并作为 `make release-check` 的依赖。测试不访问网络，也不引入 jsdom、Playwright 等前端依赖。

## 7. 文件与发布规则

- `web/dist/i18n/` 是发布产物的一部分，提交时必须和 `index.html` 一起纳入 Git。
- 一次性迁移脚本不放在 `web/dist`；本次删除了写死本机绝对路径的 `fix_i18n.py`。
- 修改翻译键时必须同时更新中英文词典、HTML/动态引用、Go 静态契约测试和本记录所述门禁。
- 不得为了绕过测试把父节点翻译标记重新挂回表单控件容器。

## 8. 验证结论

本记录对应修复完成后，必须至少执行：

```bash
make i18n-check
go test ./web ./internal/api -count=1
go test ./... -count=1
go test -race ./... -count=1
go vet ./...
```

再使用 `build-test.sh` 或等价本地服务请求 `/admin`、`/admin/i18n/index.js` 和 `/admin/api/config`，确认嵌入式 EXE 路径与测试 FS 行为一致。

本次实际验证结果：

- `make i18n-check`：通过，输出 `I18N_RUNTIME_TEST_OK`。
- `make release-check`：通过，输出 `RELEASE_CHECK_OK`；包含工具调用 smoke、govulncheck、全仓普通测试、race、vet 和 Windows amd64 交叉编译。
- macOS 本地编译二进制：`/admin` 返回 200；三个 `/admin/i18n/*.js` 均返回 200，Content-Type 为 `text/javascript; charset=utf-8`。Windows amd64 仅完成交叉编译，未在 Windows 真机浏览器执行人工点击验证。
