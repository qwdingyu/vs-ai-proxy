# Visual Studio Copilot 通用工具链第二轮协议边界审查（2026-07-15）

## 1. 背景

目标不是继续为 `create_file`、`get_file` 或某个新工具增加特判，而是让 Visual Studio Copilot 声明的任意合法工具都经过同一条可验证链路：

```text
请求 tools/functions 与历史
  -> provider 请求编码
  -> OpenAI/Ollama 上游
  -> raw/typed/SSE/NDJSON 归一化
  -> 标准 tool_calls/function_call
  -> Visual Studio 按 call id 执行并回传 tool message
```

本轮基于 `6d3c8f6` 继续审查现有矩阵测试之外的边界，重点验证“测试通过但实际仍可能偶发失败”的路径。

## 2. 发现的问题

### 2.1 provider 非流式 SSE 仍等待 EOF

`OpenAIProvider.ChatRaw` 原先对 `stream=false` 返回的 SSE 使用通用 body 读取。上游发送 `data: [DONE]` 后保持 HTTP body 打开时，代理已经拥有完整响应，却会继续等待 EOF 并触发 timeout。

### 2.2 legacy function_call 历史可能被删除

reasoning provider 的历史治理会删除“空 assistant 占位消息”。判断只看 `content/tool_calls/reasoning`，没有把 `assistant.function_call`、`refusal` 和扩展字段视为有效内容，导致旧格式工具历史丢失。

### 2.3 typed provider 响应缺少最终统一校验

部分不实现 `ChatRaw` 的 provider 直接返回 typed `ChatResponse`。非流式 handler 只做名称和 envelope 归一化，没有再次验证 `type/id/arguments/finish_reason`，非法工具可能以 HTTP 200 泄露给客户端。

### 2.4 Ollama 请求转换丢失工具控制字段

OpenAI 请求经 Ollama provider 时，原生请求构造只发送 `tools`，没有转换 legacy `functions`，也没有保留 `function_call`、`tool_choice`、`parallel_tool_calls` 和 `stop`。这会让强制工具选择、legacy 工具和停止条件在中转中消失。

### 2.5 多工具共享 ID

多个并行工具如果复用同一个 `id`，Visual Studio 后续返回的 `tool_call_id` 无法唯一关联。旧边界只检查 ID 非空，没有修复重复值。

### 2.6 finish_reason 与工具 payload 不一致

工具 payload 存在时仍返回 `finish_reason=stop`，客户端可能把完整工具当普通文本结束。工具调用必须统一为 `tool_calls`，legacy 调用必须统一为 `function_call`。

### 2.7 上游省略 [DONE]

上游给出完整 `finish_reason` 但省略 `[DONE]` 时，direct SSE 之前会返回没有标准终态的响应。代理现在在完整校验通过后补齐下游 `data: [DONE]\n\n`；没有 finish 和 DONE 的 EOF 仍然失败。

### 2.8 Ollama stop 层级错误

OpenAI `stop` 一度被保留在 Ollama 请求顶层。Ollama 原生 `ChatRequest` 把停止词定义在 `options.stop`；顶层字段可能被静默忽略，属于 JSON 中“看得到字段”但运行语义已经丢失。

### 2.9 工具终态缺少工具 payload

旧校验只约束“有 `tool_calls/function_call` 时必须使用匹配的 finish reason”，没有约束反方向。纯文本配 `finish_reason=tool_calls` 会让客户端等待不存在的工具调用。现在 raw、typed 和 direct SSE 都把没有任何工具 payload 的伪工具终态收敛为 `stop`；存在残缺工具分片时仍拒绝，不能降级掩盖截断。

### 2.10 fallback 先产生副作用且丢失第二错误

stream 转 non-stream 兜底曾在 typed 工具契约校验前设置成功诊断并写 reasoning cache；writer 拒绝响应后，缓存仍被污染。三个 alternate mode 分支还会丢弃第二次调用错误，只记录初始 503。现在先验证、成功写出后再缓存，并把“备用模式失败”和“初始模式错误”同时保留。

### 2.11 原生 Ollama raw 只校验临时投影

`/api/chat` 非流式原生路径曾把 body 转成 OpenAI typed 结构校验，却仍原样写回未校验的 raw body。`done=false`、错误 `done_reason` 或非法工具参数可能被记为成功。现在实际返回 body 必须满足 `done=true`，工具终态与 payload 一致，string arguments 转为 Ollama 对象形态，重复的非空 ID 被修复；耗时和 provider 扩展字段继续保留。

## 3. 解决方案

1. provider 成功响应使用有界增量 reader；普通 JSON 读到 EOF，SSE 在行首 `data: [DONE]` 处停止，不依赖 Content-Type 是否正确。
2. assistant 占位判断保留 `function_call/refusal/Extra`，不删除合法历史。
3. `/v1/chat/completions` 和 `/api/chat` 的 typed provider 响应在写出前统一执行完整工具契约校验。
4. Ollama provider 将非空 legacy `functions` 转为 `tools`，保留工具选择、并行调用和 legacy 强制调用，并把 OpenAI `stop` 映射到 `options.stop`；独立 converter 同步保持一致。
5. raw/typed/流式工具 envelope 对重复 ID 生成稳定代理 ID；SSE 允许合法的累计 identity 分片合并，但合并后的候选 ID 一旦与其他工具冲突，就丢弃冲突分片或生成 synthetic ID，最终下游 ID 必须唯一。
6. raw/typed/流式工具 payload 与 finish reason 必须双向匹配；没有工具 payload 的伪工具终态按普通文本 `stop` 处理，残缺 payload 仍拒绝。
7. direct SSE 在 finish 完整且校验通过后补标准 `[DONE]`，不把无终态截断流伪装成成功。
8. alternate mode 在校验和写出成功后才缓存；备用模式失败必须和初始错误一起进入诊断。
9. 原生 Ollama raw body 在实际写出前校验 `done=true`、工具参数和终态，同时保留原生扩展字段。

所有实现均使用 Go 标准库，没有增加依赖、Docker 或 Windows 专用分支。

## 4. 永久回归测试

- provider SSE 在错误 Content-Type、DONE 后保持 body 的场景必须及时返回。
- legacy `assistant.function_call` 历史在 reasoning 注入流程中必须保留。
- typed-only provider 返回非法工具 type/arguments 时必须返回 502，不能泄露调用。
- Ollama provider/converter 必须保留 legacy functions、tool choice、parallel tool calls、function call，并把 stop 放入 `options.stop`。
- raw/typed/流式重复工具 ID 必须被修复或拒绝；合法的累计 identity 分片可以合并，但最终下游 ID 必须唯一。
- direct SSE 完整 finish 但无 DONE 时补标准 DONE；无 finish/DONE 的 EOF 仍失败。
- 工具 finish 没有 payload 时不得让客户端进入空工具分支；无效 typed fallback 不得写缓存或成功诊断。
- 原生 Ollama 非流响应必须 `done=true`，实际返回 body 的 arguments/finish/ID 必须通过门禁。
- 既有固定种子随机分片、并行工具、多协议转换、DSML、截断、错误 event、5 MiB 单事件和 64 MiB 总量测试继续保留。

## 6. 剩余边界

- 代理只能保证协议转换和工具 envelope，不保证模型一定选择工具，也不能补造缺失的业务参数。
- 未收到上游响应头或首个可读流时，没有可供代理修复的工具调用；Windows 真实 `useai2/gpt-5.6-sol` 仍需现场重放。
- Windows 日志中的约 1.01 MB 请求、90 秒 `context deadline exceeded` 和 `upstream_connecting` 只能证明预算内没有得到可用上游响应，不能证明 `create_file` 未注册；`[DONE]` 早停只解决已经收到完整 SSE 后连接不关闭，不解决首响应前超时。
- DSML 仍是非标准方言，沿用有限前缀探测和大小上限；真实晚发 DSML 必须脱敏沉淀后再调整。
- Visual Studio/.NET SDK 的客户端版本差异仍需要 Windows 真机验证，wire 测试不能冒充官方客户端。

## 7. 维护规则

- 新工具只增加协议 fixture，不增加按工具名分支。
- 修改请求字段时同时检查 OpenAI provider、Ollama provider、独立 converter 和历史消息。
- 修改流式终态时同时检查 provider collector、direct SSE、OpenAI/Ollama 双向转换和 held-EOF 测试。
- 任何发布前声称“工具已修复”都必须给出 request schema、wire response、finish reason、ID、arguments 和后续 tool message 的证据。
