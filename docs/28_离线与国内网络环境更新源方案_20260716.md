# 离线与国内网络环境更新源方案

日期：2026-07-16

## 结论

短期方案应该继续保留：在客户网络不能稳定访问 GitHub 时，用 `VS_AI_PROXY_AUTO_UPDATE=0` 关闭启动自动检查，避免代理服务启动时被网络问题干扰。

产品化方案应该以 `latest.json` 静态 manifest 为核心，而不是在 Go 客户端里直接适配 GitHub、Gitee、对象存储、内网文件服务器等多个发布平台 API。这样 Go 程序只需要理解一个稳定的、供应商无关的 JSON 格式；发布流水线负责把同一批 release 产物同步到 GitHub、Gitee 或内网 HTTP 文件服务器。

Gitee 可以作为国内用户的镜像分发渠道之一，但不建议作为客户端直接依赖的“更新协议”。更合适的一体化方案是：

```text
GitHub Release 作为主发布源
        |
        | CI 生成 checksums.txt + latest.json
        v
国内镜像源：Gitee Release / 内网 HTTP / 对象存储 + CDN
        |
        v
客户端只读取 VS_AI_PROXY_UPDATE_MANIFEST_URL 指向的 latest.json
```

这条路径对当前 Go 程序改动最少，也避免把 Gitee/GitHub 的 API 差异、鉴权、限流、审核、附件 URL 变化暴露给客户安装包。

## 当前实现审查

当前更新逻辑有两个来源：

1. 默认来源：GitHub Release API。
2. 自定义来源：`VS_AI_PROXY_UPDATE_MANIFEST_URL` 或 `--update-manifest-url` 指向的静态 manifest。

当前 manifest 读取逻辑复用 GitHub Release 的最小字段：

```json
{
  "tag_name": "v0.2.53",
  "html_url": "https://intranet.example/vs-ai-proxy/v0.2.53",
  "assets": [
    {
      "name": "vs-ai-proxy-v0.2.53-windows-x64.exe.zip",
      "browser_download_url": "./vs-ai-proxy-v0.2.53-windows-x64.exe.zip"
    },
    {
      "name": "checksums.txt",
      "browser_download_url": "./checksums.txt"
    }
  ]
}
```

客户端已经支持相对 URL。也就是说 `latest.json`、Windows zip、`checksums.txt` 可以放在同一个目录：

```text
https://updates.example/vs-ai-proxy/latest.json
https://updates.example/vs-ai-proxy/vs-ai-proxy-v0.2.53-windows-x64.exe.zip
https://updates.example/vs-ai-proxy/checksums.txt
```

这个设计是正确的，因为它把“如何发布”和“如何更新”分开了：

- Go 客户端只关心版本号、资产名、下载 URL、checksum URL。
- 发布系统决定这些文件实际放在 GitHub、Gitee、Nginx、OSS、S3 还是客户内网。
- checksum 校验仍在客户端执行，下载源变化不会降低完整性校验。

## 运维建议

### 1. 完全不能联网的客户

不要启用自动更新。交付时在启动脚本或服务配置里写入：

```powershell
$env:VS_AI_PROXY_AUTO_UPDATE="0"
.\vs-ai-proxy.exe
```

适用场景：

- 客户机器不能访问公网。
- 客户网络需要人工审批软件升级。
- 客户希望升级包由 IT 部门统一投放。

这种模式下，升级流程应改为人工交付 zip 包并手动替换 exe。优点是确定、可审计；缺点是没有自动提示。

### 2. 能访问内网 HTTP/文件服务器的客户

使用静态 manifest：

```powershell
$env:VS_AI_PROXY_UPDATE_MANIFEST_URL="https://intranet.example/vs-ai-proxy/latest.json"
.\vs-ai-proxy.exe --check-update
.\vs-ai-proxy.exe --self-update
```

适用场景：

- 客户不能访问 GitHub，但能访问企业内网 HTTP 服务。
- 客户希望内部安全团队先验收版本，再开放给终端。
- 客户需要固定升级窗口。

建议目录结构：

```text
/vs-ai-proxy/
  latest.json
  checksums.txt
  vs-ai-proxy-v0.2.53-windows-x64.exe.zip
  vs-ai-proxy-v0.2.53-linux-x64.tar.gz
  vs-ai-proxy-v0.2.53-macos-arm64.tar.gz
```

`latest.json` 中推荐使用相对 URL，这样客户可以把整个目录复制到任意域名或路径，不需要改 JSON 内部地址。

### 3. 国内公网用户

推荐把 `latest.json` 和 release assets 同步到国内可访问的静态分发点，例如：

- Gitee Release。
- 企业对象存储加 CDN。
- 自有 Nginx 静态站。

其中对象存储或自有 Nginx 最稳，Gitee 更适合作为公开镜像或备用源。原因是对象存储/静态站更接近“文件分发系统”，URL 可控、缓存策略清晰、API 变化少；Gitee 是代码托管平台，Release 和仓库页面可能受到平台策略、审核、登录态、限流、附件地址形态变化等影响。

## Gitee 是否是好方案

Gitee 对国内用户有现实价值：它是国内常用代码托管平台，访问路径通常比 GitHub 更友好。Gitee 也提供官方 API 文档入口：https://gitee.com/api/v5/swagger

但从客户端更新系统角度看，Gitee 不应该成为 Go 程序里的专用更新协议。原因：

1. 当前 Go 程序已经有供应商无关的 `latest.json`。继续扩展静态 manifest 的发布端，比在客户端新增 `gitee` provider 更简单。
2. GitHub Release 和 Gitee Release 的 API、鉴权、附件上传、latest 语义、返回 JSON 字段不保证完全一致。把这些差异写进客户端，会增加长期维护面。
3. 客户环境里“能访问 Gitee”不等于“能稳定访问 Gitee API 和附件域名”。客户端直连 Gitee API 仍可能遇到 DNS、代理、证书、登录、限流或平台策略问题。
4. Gitee 适合作为镜像站，但不适合作为唯一源。企业客户更常见的可靠路径是“内部 HTTP 文件服务器”或“对象存储 + CDN”。

因此，判断是：

```text
支持 Gitee：可以。
客户端内置 Gitee API：不建议。
把 Gitee 当成 latest.json/zip/checksums 的镜像分发点：建议。
如果每次都要手工创建 Gitee Release 并手工上传 zip：暂不实施。
```

关键实施门槛：Gitee 方案必须能自动化。也就是说，GitHub tag 发布成功后，Gitee 侧至少要能通过脚本或 CI 自动完成同名 Release 创建、资产上传、`latest.json` 更新和基础校验。否则每次发布都要人工操作，步骤多、容易漏传或传错版本，还会引入 GitHub 与 Gitee 版本不一致的风险。对于当前项目，这种人工镜像流程的维护成本高于收益，暂时没有实施必要。

## 与 GitHub Release 联动的一体化方案

### 推荐方案：GitHub 主发布，流水线同步镜像

发布时以 GitHub Release 为主事实源。GitHub 官方 Release API 支持 release 和 release assets，并在响应中提供 `browser_download_url`；GitHub 官方文档也说明 release asset 可通过 `browser_download_url` 下载，并提供上传 release asset 的 API。

参考：

- GitHub Release API：https://docs.github.com/en/rest/releases/releases?apiVersion=2022-11-28
- GitHub Release Assets API：https://docs.github.com/en/rest/releases/assets?apiVersion=2022-11-28
- GitHub CLI `gh release upload`：https://cli.github.com/manual/gh_release_upload

发布流水线建议：

```text
1. 打 tag，例如 v0.2.53。
2. GitHub Actions 构建所有平台产物。
3. 生成 checksums.txt。
4. 上传 zip/tar.gz/checksums.txt 到 GitHub Release。
5. 生成 latest.json。
6. 自动把同一批文件同步到国内镜像：
   - Gitee Release，或
   - 对象存储/CDN，或
   - 内网 HTTP 文件服务器。
7. 国内 latest.json 中的 browser_download_url 使用国内镜像 URL 或相对 URL。
```

客户端无需关心第 6 步是 Gitee 还是 OSS。它只读：

```powershell
$env:VS_AI_PROXY_UPDATE_MANIFEST_URL="https://updates.example/vs-ai-proxy/latest.json"
```

如果第 6 步不能自动化，不建议把 Gitee 纳入正式发布流程。更务实的选择是先使用 GitHub Release 作为唯一公开发布源；对无法访问 GitHub 的客户，采用内网 HTTP 文件服务器或对象存储，由交付人员按客户项目节奏投放固定版本，而不是每个公开版本都手工同步 Gitee。

### Gitee 手工同步流程为何不推荐

手工流程看似简单，实际每次至少包含：

```text
1. 等 GitHub tag release 成功。
2. 到 Gitee 创建同名 Release。
3. 上传 Windows zip。
4. 上传 Linux/macOS tar.gz。
5. 上传 checksums.txt。
6. 更新或上传 latest.json。
7. 从国内网络验证 latest.json、zip、checksums.txt 都是直链。
8. 用真实 exe 执行 --check-update / --self-update。
```

这套流程的主要风险：

- 任一资产漏传，客户端会提示“发现新版本但没有匹配资产”。
- `checksums.txt` 与 zip 不匹配，会导致更新被拒绝。
- `latest.json` 先更新但 zip 还没同步完成，会让客户机器看到一个不可下载版本。
- Gitee 与 GitHub release body、附件列表、发布时间可能出现短暂或长期不一致。
- 每次发布都要人工验证，发布频率越高越不可控。

因此，Gitee 只有在“自动同步 + 自动校验”成立时才值得实施。否则 Gitee 不应进入近期方案。

### 为什么不建议客户端自动 fallback 多个平台

看起来可以让客户端按顺序尝试 GitHub、Gitee、内网源，但这会带来几个副作用：

- 每次启动多打一组外网请求，失败日志更复杂。
- 客户网络中对 GitHub 的超时可能拖慢启动。
- 不同源版本短暂不一致时，客户端可能看到 A 源最新、B 源旧版本，排障困难。
- 多源自动 fallback 会扩大测试矩阵：检查更新、下载、checksum、自更新、Windows 延迟替换都要覆盖多源组合。

更稳的做法是“环境决定唯一更新源”。客户能访问哪个源，就配置哪个 manifest URL。发布端负责保证这个 manifest 指向的文件是正确的。

## 最小 Go 改动建议

当前代码已经完成了最关键的最小改动：支持 `VS_AI_PROXY_UPDATE_MANIFEST_URL` 和 `--update-manifest-url`。

短期不建议继续改 Go 代码。先用文档和发布脚本规范流程。

如果后续确实要增强，优先级如下：

1. 增加配置文件字段，例如 `update.manifest_url`。
   - 价值：不需要客户长期设置环境变量。
   - 风险：需要管理端 UI、配置读写、文档和迁移测试。

2. 增加 `VS_AI_PROXY_UPDATE_MANIFEST_URLS` 多 URL 列表。
   - 价值：可以做显式 fallback。
   - 风险：版本一致性、超时策略、日志诊断会复杂化。
   - 建议：只有在客户现场证明单 URL 不够时再做。

3. 增加 manifest 签名。
   - 价值：即使镜像站被替换，也能校验 manifest 本身。
   - 风险：需要密钥管理、签名发布流程和客户端验签逻辑。
   - 当前已有 `checksums.txt` 校验下载包，短期可以先不做。

4. 内置 Gitee API provider。
   - 价值：客户只填 owner/repo。
   - 风险：API 差异、鉴权、限流、审核、返回字段变化、测试矩阵增加。
   - 结论：不建议作为近期方案。

5. 手工维护 Gitee Release 镜像。
   - 价值：理论上能服务部分国内用户。
   - 风险：每次发布都要人工创建 Release、上传多个资产、更新 manifest、再做下载验证；容易漏步骤。
   - 结论：不建议实施。只有自动化后再重新评估。

## 发布端实现建议

可以新增一个发布脚本，但不要让客户端依赖脚本细节。脚本职责：

```text
输入：
  VERSION=v0.2.53
  DIST_DIR=.bin/release
  MIRROR_BASE_URL=https://updates.example/vs-ai-proxy

输出：
  latest.json
  checksums.txt
  vs-ai-proxy-v0.2.53-windows-x64.exe.zip
  ...
```

`latest.json` 生成规则：

- `tag_name` 等于 Git tag，例如 `v0.2.53`。
- `html_url` 可以指向 GitHub Release 或国内镜像说明页。
- `assets[].name` 必须保持当前客户端期望的文件名。
- `assets[].browser_download_url` 推荐相对路径，例如 `./vs-ai-proxy-v0.2.53-windows-x64.exe.zip`。
- `checksums.txt` 必须包含每个 release asset 的 SHA-256。

示例：

```json
{
  "tag_name": "v0.2.53",
  "html_url": "https://github.com/qwdingyu/vs-ai-proxy/releases/tag/v0.2.53",
  "assets": [
    {
      "name": "vs-ai-proxy-v0.2.53-windows-x64.exe.zip",
      "browser_download_url": "./vs-ai-proxy-v0.2.53-windows-x64.exe.zip"
    },
    {
      "name": "checksums.txt",
      "browser_download_url": "./checksums.txt"
    }
  ]
}
```

## 客户交付模板

### 禁止启动自动检查

适合离线或严格内控环境：

```powershell
$env:VS_AI_PROXY_AUTO_UPDATE="0"
.\vs-ai-proxy.exe
```

### 使用内网更新源

适合客户能访问内网 HTTP 文件服务器：

```powershell
$env:VS_AI_PROXY_UPDATE_MANIFEST_URL="https://intranet.example/vs-ai-proxy/latest.json"
.\vs-ai-proxy.exe --check-update
.\vs-ai-proxy.exe --self-update
```

### 单次指定国内镜像源

适合临时排障：

```powershell
.\vs-ai-proxy.exe --check-update --update-manifest-url "https://updates.example/vs-ai-proxy/latest.json"
```

## 验收标准

发布镜像前至少验证：

1. `latest.json` 可被客户机器访问。
2. `latest.json` 是合法 JSON。
3. `tag_name` 是正式 release 版本，不是 `dev` 或 dirty build。
4. Windows x64 asset 名称匹配 `vs-ai-proxy-vX.Y.Z-windows-x64.exe.zip`。
5. `checksums.txt` 可访问。
6. 客户机器执行 `--check-update` 能看到新版本。
7. 客户机器执行 `--self-update` 后，新 exe 能启动并输出目标版本。

## 风险与边界

- 自动更新只解决代理程序自身升级，不解决 Visual Studio Copilot 客户端版本、Windows 网络策略或上游 provider 可用性。
- `VS_AI_PROXY_AUTO_UPDATE=0` 只关闭启动自动检查；用户仍可手动执行 `--check-update` 或 `--self-update`。
- `latest.json` 是“当前推荐版本”的指针。发布端必须保证它最后更新，不能先更新 manifest 再上传 zip/checksum。
- 镜像同步失败时，不要更新 `latest.json`。
- Gitee 镜像要作为“可访问性优化”，不要作为唯一可信源。可信性仍来自 release 产物、checksum、发布流程和权限控制。

## 建议决策

近期采用：

```text
默认：GitHub Release
离线客户：VS_AI_PROXY_AUTO_UPDATE=0
国内/企业客户：VS_AI_PROXY_UPDATE_MANIFEST_URL 指向内网或国内 latest.json
发布侧：GitHub Release 后自动生成并同步 latest.json + assets + checksums.txt
```

暂不做：

```text
Go 客户端内置 Gitee API
Go 客户端自动探测 GitHub/Gitee 多源
Go 客户端默认改成 Gitee
```

这个决策能保持客户端稳定，把国内网络适配放在发布和运维层解决，是当前风险最低、代码改动最少、可解释性最强的方案。
