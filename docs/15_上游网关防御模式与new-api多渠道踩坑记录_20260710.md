# 上游网关防御模式与 new-api 多渠道踩坑记录（2026-07-10）

## 背景

本次问题发生在 Visual Studio / Copilot BYOM 真实使用链路中，主要集中在 `UseAI` / `useai2` 的 `gpt-5.5`，以及类似 new-api、sub2api 的 OpenAI-compatible 网关。

用户侧看到的典型现象包括：

- `/v1/models` 可以返回 `gpt-5.5`，但 `/v1/chat/completions` 失败。
- 同一模型有时 2 到 4 秒返回，有时 30 到 40 秒返回，有时 60 秒以上超时。
- 上游返回 `500 do_request_failed`、`503 Service temporarily unavailable`、`429 rate limited`、`context deadline exceeded`。
- 客户端发送 `stream=false`，上游仍返回 `data: ...` SSE，导致非流式 JSON 解析失败。
- 默认 Python/Go User-Agent 访问 `/models` 可能被 WAF 拦截，但带稳定 UA 正常。
- Visual Studio/Copilot 工具调用里出现 `create_file`、PowerShell、Git 等工具无法运行，根因可能是工具调用协议字段或流式分片被破坏。

这类问题不能只按“某个模型坏了”处理。它本质上是代理、OpenAI-compatible 网关、new-api 内部渠道和 Visual Studio/Copilot 严格客户端之间的边界兼容问题。

## 已确认事实

1. new-api 公开 README 宣称支持 `Channel weighted random` 和 `Automatic retry on failure`。
2. 这只能说明 new-api 内部有渠道选择/重试机制，不代表每次失败都能对下游透明兜住。
3. 实测 `do_request_failed`、503、429、超时、SSE 非标准返回都会透出到 vs-ai-proxy。
4. `/v1/models` 成功只代表模型列表接口可读，不代表 chat completions 一定可用。
5. Visual Studio/Copilot 对 OpenAI 协议比浏览器调试更严格：非法 `finish_reason`、错误 SSE、缺失/变形工具字段都可能导致功能失败。

## 关键误判

### 误判 1：new-api 内部有多渠道，所以代理不用处理失败

错误。new-api 内部重试是第一层保护，但它失败时仍会把错误透出。代理必须把它当成“不完全稳定的 OpenAI-compatible 网关”，而不是当成强一致、高可用 OpenAI 官方端点。

### 误判 2：代理侧无限重试可以解决问题

错误。无限或长等待重试会导致：

- Visual Studio 请求卡死。
- 用户重复点击或取消后，代理仍继续消耗上游额度。
- 429 限流被放大。
- 非幂等工具调用场景可能产生重复副作用。

代理侧只能做短重试和冷却排序，不能替代上游网关的渠道治理。

### 误判 3：关闭防御只是不重试 provider 内部请求

错误。代理还有候选 provider fallback、流式/非流式互相兜底、动态模型 catalog 等机制。关闭防御时必须一起关闭这些会产生第二次上游请求或掩盖原始错误的行为。

本次新增测试就抓到了这个问题：即使关闭 provider 内部重试，同一请求仍可能因为候选列表重复或跨 provider fallback 造成两次上游访问。

## 最终策略

新增配置：

```json
{
  "defense": {
    "enabled": true
  }
}
```

默认值：开启。

设计原因：普通 Windows / Visual Studio 用户更需要稳定默认行为；关闭只用于高级排障和复现上游原始行为。

### 防御开启时

- 给 OpenAI-compatible 上游发送稳定 `User-Agent: vs-ai-proxy`。
- 发送 `X-Requested-With: vs-ai-proxy`，便于 WAF 和日志识别。
- 对 5xx、EOF、连接重置、empty reply、瞬态 stream error 做最多 3 次短重试。
- 不重试 400、401、403、404、429。
- 429 记录 provider 冷却，优先尊重 `Retry-After`，最多冷却 10 分钟。
- 同一模型多个 provider 候选时，失败 provider 会短时间排到候选末尾。
- 非流式失败可尝试流式聚合为 JSON。
- 流式失败且尚未写出响应时，可尝试非流式并合成为 SSE。
- `stream=false` 却收到 SSE 时，聚合成标准 `chat.completion` JSON。

### 防御关闭时

- provider 内部不做短重试。
- 不发送防御用 User-Agent / X-Requested-With。
- 不做流式/非流式互相兜底。
- 不做跨 provider 自动 fallback，只保留首选候选。
- 用于观察上游原始错误、请求次数、耗时和响应体。

## 核心代码位置

- 配置结构：`internal/config/config.go`
- Web 开关：`web/dist/index.html`
- provider 请求防御：`internal/provider/provider.go`
- provider 健康、冷却和候选排序：`internal/provider/registry.go`
- 真实 `/v1/chat/completions` 防御开关接入：`internal/proxy/server.go`
- SSE 聚合、finish_reason、工具调用归一化：`internal/proxy/openai_chat_response.go`
- 管理测试页流式兜底：`internal/api/api.go`

## 本次修复点

### 1. 防御模式可配置

Web 管理端 `/admin#/config` 新增“启用上游防御模式”。保存后写入 `config.json`，并热更新代理服务。

旧配置没有 `defense` 字段时，运行时默认补为开启，避免升级后保护失效。

### 2. 稳定 User-Agent

防御开启时，OpenAI-compatible 请求统一携带：

```http
User-Agent: vs-ai-proxy
X-Requested-With: vs-ai-proxy
```

如果确实需要自定义 UA，可通过环境变量覆盖：

- `VS_AI_PROXY_USER_AGENT`
- `PROVIDER_USER_AGENT`
- `OPENAI_COMPAT_USER_AGENT`

### 3. Retry-After 冷却

如果上游返回：

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 120
```

代理不会在当前请求里等待 120 秒重试，而是把 provider 置入约 120 秒冷却。后续同模型如有其他 provider 候选，会优先走其他候选。

原因：Visual Studio/Copilot 是交互式请求，当前请求长时间等待会造成用户误以为卡死。

### 4. 候选去重

静态配置和动态 `/models` catalog 可能同时把同一个 provider/model 放入候选。现在在排序前去重，避免一次用户请求重复打到同一个 provider。

### 5. 防御关闭时限制候选

防御关闭用于诊断原始上游问题，因此只尝试首选候选，不做跨 provider 自动兜底。否则用户看到的成功/失败可能不是当前 provider 的真实行为。

### 6. SSE 空 choices 兼容

部分 new-api 网关会返回 usage-only SSE chunk，例如 `choices: []`。非流式转换时不能因此把 `data:` 原文泄漏给下游。当前逻辑允许聚合为空内容的标准 JSON，避免 JSON 解析失败。

## 测试覆盖

本次新增/确认的测试场景：

- 旧配置升级后 `defense.enabled` 默认开启。
- 显式 `defense.enabled=false` 会被保存并保留。
- Web/API 保存配置后，代理热更新 provider 防御模式。
- 防御关闭后，chat 请求不做 provider 内部重试。
- 防御关闭后，不发送 `vs-ai-proxy` 防御 Header。
- 429 `Retry-After` 会转为 provider 冷却。
- 候选去重避免重复请求同一 provider/model。
- 非流式收到 SSE 能转换为 JSON。
- 空 choices SSE 不再导致 `data:` 文本泄漏给非流式下游。

## 仍需人工实机验证

自动化测试覆盖了协议和代理行为，但不能完全替代 Windows + Visual Studio/Copilot 实机验证。

发版前建议至少人工验证：

1. Windows 启动时端口占用清理是否符合预期。
2. `/admin#/config` 开关保存后，重启和热更新都能保留配置。
3. `useai2 + gpt-5.5` 的 `stream=true` 和 `stream=false` 都能返回可解析结果，或给出清晰上游错误。
4. VS Copilot 创建文件、运行 PowerShell、运行 Git、读取文件工具调用均能产生可见修改痕迹。
5. 防御关闭后，日志展示的是原始上游错误，不被 fallback 掩盖。

## 后续维护原则

1. 不要针对单个模型名硬编码规则，例如只写 `gpt-5.5` 特判。
2. 所有 OpenAI-compatible 网关兼容都应沉到 provider/proxy 公共层。
3. 新增 fallback 时必须回答三个问题：
   - 是否可能产生第二次上游请求？
   - 是否可能重复执行工具调用副作用？
   - 是否应该受 `defense.enabled` 控制？
4. 任何测试页兜底都必须同步检查真实代理路径，避免“测试页成功、VS 失败”。
5. 429 不做立即重试，优先冷却；5xx/网络瞬断才允许短重试。
6. 不能相信 `/models` 成功等价于 chat 成功。
7. 不能相信 new-api 内部多渠道等价于下游永远不会收到 5xx。
