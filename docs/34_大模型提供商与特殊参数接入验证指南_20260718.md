# 大模型提供商与特殊参数接入验证指南（2026-07-18）

本文是新增或验证大模型 provider、特殊模型和特殊参数时的操作说明书。目标是防止为了修一个 provider 在核心 handler 里不断打补丁，最终破坏 Visual Studio / Copilot 的主链路。

核心原则：

1. **先验证事实，再改代码**：不能凭“OpenAI-compatible”假设所有参数语义相同。
2. **先找归属层，再决定改动面**：provider 方言、模型能力、请求治理和协议 adapter 是四种不同问题。
3. **默认不动 `/v1/chat/completions` handler**：handler 只负责路由、fallback、诊断和最终响应，不承载单 provider 知识。
4. **所有特殊性必须可复现、可测试、可文档化**：临时 curl 结论不能替代仓库测试。

## 1. 系统扩展边界

| 层级 | 事实源 | 适合放什么 | 不适合放什么 |
| --- | --- | --- | --- |
| Provider capability | `internal/provider/capabilities.go` | API format、chat/models path、默认 base URL、env 前缀、是否支持 `reasoning_effort` / `top_k`、输出 token 参数名 | 单个模型上下文长度、模型是否支持工具、业务 prompt 规则 |
| Model profile | `internal/provider/model-selection/*.json` 和 `internal/provider/model_catalog.go` | 模型名、别名、context length、max output、工具/视觉能力、推荐 temperature/top_p/max_tokens/reasoning_effort、timeout | provider 通用 path、provider 鉴权方式、所有模型共享的参数方言 |
| Request transformer | `internal/proxy/request_transformer.go` | 按 provider capability 和 model profile 合并/剥离请求参数，保护工具请求不被默认输出上限截断 | 解析上游响应、处理 SSE chunk、硬编码某个 provider 的 URL |
| Provider adapter | `internal/provider/provider.go`、Ollama 相关 adapter | 请求序列化、URL 拼接、header、原生协议字段映射、usage 读取 | Visual Studio 专属工具兼容、候选 fallback 策略 |
| Proxy protocol layer | `internal/proxy` 的 SSE/工具/DSML 处理 | VS/Copilot 协议兼容、工具调用完整性门禁、fallback、诊断日志 | provider 参数方言表、模型能力数据库 |

判断规则：

- 如果差异对同一 provider 的多个模型都成立，优先放 **Provider capability**。
- 如果差异只属于某个模型或模型族，优先放 **Model profile**。
- 如果差异是“客户端字段如何合并/剥离”，放 **Request transformer**。
- 如果差异是“上游协议不是 OpenAI Chat Completions 的 JSON/SSE 形态”，才考虑 **Provider adapter**。
- 只有当差异无法被上面四层表达时，才评估修改 proxy protocol layer。

## 2. 禁止事项

禁止以下做法：

1. 在 `/v1/chat/completions` handler 中写 `if model == "xxx"` 或 `if provider == "xxx"` 来改请求参数。
2. 为了让统计数字好看，把 missing usage 当成 0。
3. 在不确定上游是否支持的情况下主动注入 `reasoning_effort`、`top_k`、`parallel_tool_calls`、`store`、`metadata` 等参数。
4. 对工具请求无条件注入 `max_tokens=4096` 或 `temperature=0.7`，这会截断大工具参数。
5. 在下游 SSE 已经开始写出后切换 provider。
6. 把 provider 账单、代理 token 统计和模型估算 token 混成一个数字。
7. 只新增文档不新增测试，或只新增测试不补文档。

## 3. 新 provider 接入流程

### 3.1 最小事实采集

对每个 provider 先收集这些事实：

| 事实 | 验证方式 | 记录位置 |
| --- | --- | --- |
| base URL 是否已带 `/v1` | 用真实 `/models` 和 `/chat/completions` 请求验证 | 踩坑文档或接入记录 |
| chat path | 非流式最小请求 | `ProviderCapabilities.ChatPath` |
| models path | 官方模型列表请求 | `ProviderCapabilities.ModelsPath` |
| 输出 token 字段 | 对比 `max_tokens`、`max_completion_tokens`、`max_output_tokens` | `ProviderCapabilities.OutputTokenParam` |
| `reasoning_effort` | 分别带/不带该字段请求 | `ProviderCapabilities.SupportsReasoningEffort` |
| `top_k` | 分别带/不带该字段请求 | `ProviderCapabilities.SupportsTopK` |
| 流式 SSE 终态 | 验证 `[DONE]`、finish_reason、usage chunk | provider/proxy 流式测试 |
| 工具调用 | 最小 weather 工具 + 多轮 tool result | proxy 工具测试 |
| usage | 非流、流式、Ollama 原生终态 | store/proxy token 测试 |

### 3.2 最小 curl 矩阵

至少验证：

```bash
# 1. 模型列表
curl -sS "$BASE_URL/$MODELS_PATH" -H "Authorization: Bearer $API_KEY"

# 2. 非流式普通聊天
curl -sS "$BASE_URL/$CHAT_PATH" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"MODEL","messages":[{"role":"user","content":"reply OK"}],"stream":false,"max_tokens":32}'

# 3. 流式普通聊天
curl -N "$BASE_URL/$CHAT_PATH" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"MODEL","messages":[{"role":"user","content":"reply OK"}],"stream":true,"max_tokens":32}'

# 4. 工具调用
curl -sS "$BASE_URL/$CHAT_PATH" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"MODEL","messages":[{"role":"user","content":"weather in Shanghai"}],"tools":[{"type":"function","function":{"name":"get_weather","parameters":{"type":"object","properties":{"city":{"type":"string"}},"required":["city"]}}}],"tool_choice":"auto","stream":false,"max_tokens":256}'
```

如果 provider 涉及推理模型，必须额外对比：

```text
max_tokens=32
max_completion_tokens=32
max_output_tokens=32
```

对比项：

- HTTP status；
- 是否返回 content；
- `finish_reason` 是否为 `stop` / `tool_calls` / `length`；
- 是否出现 `reasoning_content`；
- usage 中 `reasoning_tokens` 是否被计入 total；
- 工具参数是否完整 JSON。

## 4. 修改指南

### 4.1 只改 provider capability 的情况

适用：

- path 不同；
- base URL 不同；
- provider 是否支持 `reasoning_effort` / `top_k` 不同；
- 输出 token 参数名不同，例如 MiMo 使用 `max_completion_tokens`。

改动点：

1. `internal/provider/capabilities.go` 增加或修改 provider entry。
2. 如需自动识别，补 `InferCapabilityName()` 的 base URL 或 ID 规则。
3. 补 `internal/provider/capabilities_test.go`。
4. 补 `internal/provider/provider_test.go`，验证请求 body 的字段名/path/header。
5. 如果管理页需要展示，只通过 `CompatibilityProfileFor()` 暴露，不在 Web 端重复推断。

必跑：

```bash
go test ./internal/provider ./internal/api ./web -count=1
```

### 4.2 只改 model profile 的情况

适用：

- 新增模型名、别名；
- 上下文长度或最大输出；
- 模型支持工具/视觉；
- 某个模型推荐 `reasoning_effort`、temperature、top_p；
- 某个模型需要更长 timeout。

改动点：

1. `internal/provider/model-selection/<provider>.json`。
2. 必要时更新 `internal/provider/model-metadata/models.json`。
3. 补 `internal/provider/model_catalog_test.go` 或相关模型发现测试。

必跑：

```bash
go test ./internal/provider ./internal/proxy -run 'Model|Profile|Transform|Show' -count=1
```

### 4.3 需要改 request transformer 的情况

适用：

- 参数需要按 provider capability 条件剥离；
- 工具请求和普通聊天的默认值策略不同；
- profile 和客户端参数优先级需要调整。

准入要求：

- 必须证明现有 capability/model profile 无法表达；
- 必须有“普通聊天”和“工具请求”两组测试；
- 必须验证不会恢复 `max_tokens=4096` 截断工具参数的问题。

必跑：

```bash
go test ./internal/proxy -run 'Transform|Tool|Contract' -count=1
go test ./internal/proxy -run 'Copilot|Tool' -count=1
```

### 4.4 需要新增/修改 provider adapter 的情况

适用：

- 上游不是 OpenAI Chat Completions / Ollama 的 JSON 或 SSE 形态；
- usage 在非标准字段；
- 工具调用字段不在 OpenAI `tool_calls` 或 legacy `function_call` 中；
- 需要 provider 专属 header。

准入要求：

- 先尝试用 `ProviderCapabilities` 表达 path/参数差异；
- 不能因为一个参数名不同就新增 adapter；
- adapter 必须同时覆盖非流、流式、错误分类和 usage。

必跑：

```bash
go test ./internal/provider ./internal/proxy ./internal/converter -count=1
```

## 5. 测试准入矩阵

新增 provider 或特殊模型发布前，至少通过：

| 风险 | 必测项 |
| --- | --- |
| 路径拼接 | base URL 带 `/v1` 和不带 `/v1` 都不能出现 `/v1/v1` |
| 非流式 JSON | content、tool_calls、finish_reason、usage |
| 流式 SSE | 多行 data、usage-only chunk、`[DONE]`、截断错误 |
| 工具调用 | modern `tool_calls`、legacy `function_call`、工具参数完整 JSON |
| 大上下文 | 大请求体、timeout budget、499/client canceled 诊断 |
| 参数方言 | `max_tokens` / `max_completion_tokens` / `max_output_tokens` 收敛目标 |
| reasoning | `reasoning_content` 保留/剥离、`reasoning_effort` 是否透传 |
| usage | unknown 不当 0、reported zero 保留、cached/reasoning 是子集 |
| 日志 | 成功/失败都带 request_id/provider/requested/upstream |
| Web | compatibility profile 展示与中英文 i18n |

发布前建议命令：

```bash
go test ./... -count=1
go test -race ./internal/provider ./internal/proxy ./internal/store -count=1
go vet ./...
GOOS=windows GOARCH=amd64 go build -o /tmp/vs-ai-proxy-windows-check.exe ./cmd/server
```

如果改了 Web：

```bash
go test ./web -count=1
```

如果改了工具协议：

```bash
go test ./internal/proxy -run 'Tool|Copilot|DSML|Stream|Function' -count=1
```

## 6. 日志判断规则

控制台中一条 provider warning 不等于最终请求失败。

正确判断顺序：

1. 先按 `request_id` 归并；
2. 再看 `provider`、`requested_model`、`upstream`；
3. 再看最终 `POST /v1/chat/completions - <status>`；
4. 最后看 `attempts` 中是否有 fallback。

典型解释：

| 现象 | 解释 |
| --- | --- |
| `mimo` warning 后又有 `200` | 候选失败后可能有最终成功，必须看同一 request_id |
| MiMo 会话中出现 `step-router` warning | 可能是并发请求、默认模型旧会话、未强绑定 provider 或 fallback 候选 |
| `上游响应无法识别` | 上游返回了内容，但 JSON/SSE/工具/终态契约不安全 |
| `无法连接上游` | 连接、TLS、网关、WAF、等待响应头等网络阶段失败 |

## 7. 代码审查清单

合并前逐项确认：

- 是否把 provider 差异放到了 `ProviderCapabilities`。
- 是否把模型差异放到了 model-selection JSON。
- 是否避免在 handler 中写模型名/provider 名特判。
- 是否有 unknown usage 和 reported zero 的测试。
- 是否有 stream 和 non-stream 两条测试。
- 是否有工具请求测试，尤其大工具参数不被截断。
- 是否更新 `CompatibilityProfileFor()` 或 Web 展示。
- 是否更新 docs，对原因、排除项、风险和验证命令有记录。
- 是否跑过全量测试、race、vet、Windows build。

## 8. 当前仍可优化但不建议立刻扩大范围

这些方向有价值，但不应混入单个 provider 修复：

1. **Provider capability 字段进一步结构化**：例如把 unsupported parameters、required headers、usage source 做成 declarative rules。需要整体设计和迁移测试，不应为单模型临时加。
2. **真实 provider smoke harness**：把 curl 矩阵变成可配置脚本，但不能要求 CI 使用真实 API key。
3. **Web 上按 provider/model 筛选周期 token**：当前首页先做全局日/周/月；筛选维度应作为独立小迭代，避免一次改动过大。
4. **跨时区远程管理页语义**：本项目默认本机代理，本地时区足够；如果未来支持远程多人访问，再把 period key 和展示时区做成显式配置。

以上优化都不能破坏当前核心目标：本地代理稳定、VS/Copilot 协议保真、失败可诊断、统计不伪造。
