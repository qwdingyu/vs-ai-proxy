# Windows 大上下文超时与工具协议闭环排查（2026-07-14）

## 1. 背景

Windows 真实用户环境出现两类现象：

1. Visual Studio Copilot 偶尔无法执行 `create_file` 等已声明工具。
2. `useai2/gpt-5.6-sol` 在约 1.01 MB 请求下等待约 90 秒后返回 `502 timeout`。

典型日志：

```text
模型 gpt-5.6-sol 在提供商 useai2 流式失败:
Post "https://api.eforge.xyz/v1/chat/completions": context deadline exceeded

POST /v1/chat/completions - 502 (90375 ms)
error_code=timeout
request_bytes=1007.6 KB
upstream_bytes=1.01 MB
stream_state=upstream_connecting
attempts="useai2/gpt-5.6-sol 90s timeout"
```

该日志来自 Windows，不是本机进程。本机日志和本机旧二进制只能用于验证开发环境，不能替代 Windows 样本，也不能反推 Windows 当时运行的版本。

## 2. 证据边界

### 2.1 日志已经证明的事实

- 请求已完成代理路由，目标是 `useai2/gpt-5.6-sol`。
- 客户端请求体约 1007.6 KB，代理治理后的上游请求约 1.01 MB，代理没有出现数量级膨胀。
- 有效请求预算是 90 秒。
- `stream_state=upstream_connecting` 表示 `ChatStream` 尚未拿到上游 HTTP 响应头并建立可读响应流。
- 代理在预算耗尽后返回 502，没有收到可验证的完整工具调用。

`upstream_connecting` 不是“只在做 DNS/TCP 连接”。这个阶段可能包含请求体上传、DNS/TCP/TLS、CDN/WAF、上游网关排队/选渠道，以及上游直到首个 SSE 数据才发送响应头的等待时间。没有更低层 trace 时，不能只选其中一项写成根因。

### 2.2 日志不能证明的内容

- 不能证明 `create_file` 没有注册。请求日志已经声明工具，但上游流未建立。
- 不能证明上游已返回完整 `tool_calls` 后被代理丢失。该请求没有原始上游 SSE fixture。
- 不能仅凭 1.01 MB 判断超过模型 token 上限。字节数不是 token 数，工具 schema、代码和自然语言的 token 密度不同。
- 不能从本机运行版本推断 Windows 运行版本。复现时必须同时记录 `--version` 或启动版本日志。

因此，该条 90 秒日志的准确结论是：

> 大上下文请求在有效客户端预算内没有建立上游响应流，工具调用未能完整返回是超时结果，不是工具注册失败的证据。

## 3. 全链路审查结果

本次没有继续给 `create_file` 增加模型/provider 特判，而是按所有工具共用的协议入口审查：

```text
OpenAI raw JSON
OpenAI direct SSE
stream=false -> SSE 聚合
stream=true -> 非流式 fallback
legacy function_call
DSML 文本方言
Ollama JSON / NDJSON
```

### 3.1 确认缺陷一：alternate mode 丢 legacy 调用

当流式请求在尚未写下游前遇到可恢复 5xx，代理会在同一 provider 内尝试非流式。非流式响应若使用 legacy `function_call`，原 SSE 合成器只输出 `finish_reason=function_call`，没有输出 `delta.function_call`。

客户端因此看到“工具调用结束”，却没有工具名和参数，表现为偶发无法执行工具。

### 3.2 确认缺陷二：截断 DSML 绕过统一保护

标准 `tool_calls` 已禁止在 `length/content_filter` 下执行，但 DSML 转换发生在该保护之后。截断响应中的 DSML 文本仍可能被重新转换为可执行工具。

这不是单个工具名问题，而是终态规则没有覆盖所有协议方言。

### 3.3 确认缺陷三：OpenAI 转 Ollama 吞 SSE 错误

OpenAI SSE 转 Ollama NDJSON 时，解析错误原先被 `continue` 跳过，循环末尾仍发送 `done=true`。上游工具错误可能因此变成下游的静默成功。

同类问题还会使 Ollama handler 在已经写出部分响应后把 provider 记为成功，污染后续健康状态。

## 4. 排查过程

1. 先校正样本来源，明确 Windows 远端日志与本机运行进程无关。
2. 按 `stream_state` 的实际设置位置追踪：调用 `ChatStream` 前是 `upstream_connecting`，拿到 HTTP 响应后才是 `upstream_connected`。
3. 核对配置链：默认 `client_timeout_budget_seconds=90`，模型/profile 的更长超时会被裁剪到客户端预算，最大允许 95 秒。
4. 核对路由链：稳定模式只执行首选 provider，避免跨 provider 重复计费、模型混用和工具副作用，因此日志只有一次 `useai2` 尝试符合当前设计。
5. 对最近的工具协议统一改动逐入口构造正式测试，不使用验证后删除的临时 overlay。
6. 修复前测试稳定复现 legacy 调用丢失、截断 DSML 误执行和 OpenAI-to-Ollama 错误伪成功；修复后三组测试通过。

## 5. 解决方案

### 5.1 保留 legacy `function_call`

统一 SSE 合成器同时输出：

- modern `delta.tool_calls`；
- legacy `delta.function_call`；
- 与结构匹配的 `finish_reason`。

这覆盖 stream-to-nonstream fallback，不按 `create_file` 名称写特判。

### 5.2 统一截断不执行规则

`length/content_filter` 现在对以下入口使用同一条不变量：

- OpenAI typed response；
- OpenAI raw JSON；
- SSE 聚合后的响应；
- DSML 文本转换；
- Ollama raw JSON。

只要响应被截断，就不能把任何方言重新转换为可执行工具调用。

### 5.3 错误必须保持失败语义

OpenAI SSE 转 Ollama 遇到错误帧或无效 JSON 时立即返回协议错误，不再补发虚假的 `done=true`。已经写出部分下游响应时不能切换候选，但 provider 仍记录为失败。

### 5.4 对 90 秒超时不做危险改写

本次没有：

- 伪造模型没有返回的 `create_file`；
- 静默删除 1 MB 历史；
- 在下游已经开始后混入其他 provider 的流；
- 把超时预算无限提高到 VS/Copilot 客户端等待窗口之外；
- 对所有大请求自动 gzip，因为 OpenAI-compatible 网关不保证接受压缩请求体。

这些方案会改变语义、产生重复计费或制造新的兼容问题，不能在缺少真实 fixture 时冒险上线。

## 6. 正式测试

新增并保留以下回归场景：

1. 流式 503 后同 provider 非流式 fallback 返回 legacy `create_file function_call`，最终 SSE 必须包含完整工具名、参数和 `finish_reason=function_call`。
2. OpenAI typed/raw 与 Ollama raw 的 DSML 在 `length/content_filter` 下均不得生成 `tool_calls`。
3. OpenAI-to-Ollama 收到 SSE error 时必须返回错误，且不得发送 `done=true`。

工具协议修改后最低验证：

```bash
make tool-check
go test ./internal/proxy ./internal/provider ./internal/converter -count=1
go test -race ./internal/proxy ./internal/provider ./internal/converter -count=1
go vet ./...
make release-check
```

还必须构建 Windows x64、macOS x64 和 macOS arm64。项目不依赖 Docker，不能用容器结果替代桌面平台产物验证。

## 7. Windows 复测步骤

1. 使用包含本次修复的 Windows x64 正式构建，先记录 `vs-ai-proxy.exe --version`。
2. 新 session 发起小上下文 `create_file`，记录 request ID、`request_tools`、`response_tools`、`finish_reason` 和 `stream_state`。
3. 原 session 或约 1 MB fixture 发起同工具请求，记录同一组字段。
4. 如果仍是 `upstream_connecting + 90s timeout`，应查 `useai2` 上游网关的渠道排队、首响应时间和大上下文能力；代理没有收到响应可供修复。
5. 如果是 HTTP 200 且 `response_tools` 包含 `create_file`，但 VS 仍无法执行，应保存脱敏后的完整 SSE/JSON 和客户端错误，继续检查工具名、arguments schema 与客户端解析。
6. 如果 `response_tools` 为空但上游已 200 完成，应保存原始上游响应，判断模型没有调用工具、返回 DSML/私有方言，还是代理转换缺口。

真实上游矩阵会产生计费，必须使用用户明确提供的测试 key/额度执行；本地单元测试不能冒充真实渠道验证。

## 8. 落地建议

- 1 MB 旧 session 超时而新 session 恢复时，优先减少历史、附件和打开文件，而不是继续补工具别名。
- 工具任务优先选择已有 `response_tools` 成功记录的 provider/model/channel。
- 如果上游网关内部有多渠道，单渠道超时应留出在 90 秒总预算内切换的空间；本地代理不替代上游渠道调度。
- 保留 `client_timeout_budget_seconds=90` 的默认值。可在确认 VS 客户端实际等待更久后谨慎调到 95，但不能期待额外 5 秒解决持续排队或超大上下文能力不足。
- 下一次同类问题必须提供版本、request ID 和脱敏原始响应；只有日志摘要时必须明确结论边界。

## 9. 剩余风险

- 当前没有该 Windows request ID 对应的脱敏原始上游 SSE，因此不能证明真实 `gpt-5.6-sol` 在超时前是否产生过任何内容。
- 8 KiB 之后才首次出现的 DSML 保持现有延迟权衡：代理不会为未知方言无限缓存全部普通流。出现真实样本后再用正式 fixture 调整。
- 上游首响应超过客户端预算无法由协议归一化修复，只能通过会话减压、稳定渠道或上游调度改善。
