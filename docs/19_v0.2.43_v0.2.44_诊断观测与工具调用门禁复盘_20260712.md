# v0.2.43 / v0.2.44 诊断观测与工具调用门禁复盘（2026-07-12）

## 1. 背景

v0.2.42 解决了 VS Copilot 工具调用主链路的稳定透传问题，但发布前后仍暴露出两个更高层面的风险：

1. 用户和客服看到 413、499、503、timeout、context canceled 时，仍需要人工猜测到底是客户端等待上限、上游首 token 慢、网关 5xx、网络连接断开，还是请求体过大。
2. 工具调用虽然本地验证通过，但如果没有固定门禁，后续快速发版时仍可能因为漏跑某条协议路径而回归。

本轮目标不是继续对某个错误码打补丁，而是从“可观测性”和“发布门禁”两个方向做收敛：

- 让管理页日志、服务端日志、上游网关日志能用同一个 `request_id` 串起来。
- 让最近失败、候选耗时、结构化诊断在管理页前置展示。
- 把工具调用专项测试固化为 `make tool-check` 和 GitHub Actions 门禁。
- 记录 Windows 自更新、端口占用、前端单文件漂移等高风险路径的状态和检测方式。

## 2. v0.2.43 核心变更

### 2.1 结构化诊断字段

请求日志新增/强化以下字段：

- `request_id`：贯穿管理页、服务端日志和上游请求头。
- `error_reason`：客服可读的一句话原因。
- `error_action`：下一步处理建议。
- `diagnostic_summary`：压缩展示耗时、请求体、上游体、远端 peer、流状态。
- `attempts_summary`：多候选 provider/model 的单次耗时和错误分类。

这些字段的目的不是替代原始错误，而是让用户一眼知道优先排查哪一层。

### 2.2 错误分类进一步细化

本轮重点区分：

| 分类 | 典型含义 | 排障方向 |
| --- | --- | --- |
| `client_deadline_reached` | VS/Copilot 接近约 100 秒等待上限后取消 | 查上游首 token、new-api/sub2api 单渠道超时和轮换 |
| `client_gone` | 用户取消、窗口关闭或短耗时客户端断开 | 短耗时通常不是代理问题 |
| `timeout` | 代理按有效预算主动结束慢请求 | 查模型 `timeout_seconds` 和上游首 token |
| `upstream_payload_too_large` | 413 或等价大请求拒绝 | 查上下文、工具声明大小、provider/channel 限制 |
| `upstream_server_error` | 500/502/503/504 | 查上游网关渠道健康和重试/轮换 |
| `upstream_rate_limit` | 429 | 查额度、并发、冷却 |
| `upstream_request_error` | 400/404 | 查模型名、provider 绑定、参数治理 |
| `network_error` | DNS/CDN/Cloudflare/WAF/连接重置 | 查网络链路和远端 peer |
| `proxy_parse_error` | HTML、SSE/JSON 不匹配或解析失败 | 保存响应样本排查协议 |

### 2.3 最近失败样本不能只看当前页

复审发现一个容易误导客服的问题：日志页如果只在当前页或最近固定条数里找失败，连续健康检查成功后可能把真正有价值的失败样本隐藏掉。

修复方式：

- Store 层新增 `GetLatestFailure()`。
- `/api/diagnostics/summary` 使用 Store 中当前保留日志里的最近失败。
- 日志页最近失败卡片优先读取统一诊断摘要里的全局 `latest_failure`。

对应测试覆盖：

- 失败日志在较早位置，后面连续 60 条成功日志，仍能找回最近失败。
- 诊断摘要的 `copy_summary` 仍包含这条失败的 `request_id`。

### 2.4 统计并发锁修复

复审发现 Store 统计存在潜在数据竞争：

- `AddLog()` 会写 `TotalRequests`、`SuccessCount`、`FailureCount`、`AvgLatencyMs`。
- `GetStatistics()` 会并发读取这些字段。
- 如果写路径只持有 `logMu`，而读路径持有 `statsMu`，race 检测可能暴露并发风险。

修复方式：

- `AddLog()` 更新统计时同时持有 `statsMu`。
- 补充 `TestStoreConcurrentAddLogAndStatistics`。
- `go test -race ./internal/store ./internal/api` 已覆盖。

### 2.5 统一 request metadata 层

最初实现 request_id 透传时，proxy 和 provider 各自维护 context key，这会导致后续扩展 `trace_id` / `span_id` 时继续分裂。

本轮收敛为：

- 新增 `internal/requestmeta`。
- `requestmeta.Metadata` 当前包含 `RequestID`。
- proxy middleware 生成或接收 request_id 后写入 request metadata。
- provider 层从统一 metadata 读取 request_id。
- OpenAI-compatible 和 Ollama 上游请求都会带：
  - `X-Request-ID`
  - `X-Proxy-Request-ID`

设计原则：

- 不在 provider 层重新生成 ID。
- 不让 proxy/provider 各自维护独立 context key。
- 后续如果需要 `trace_id`、`span_id`、`tenant_id`，优先扩展 `requestmeta.Metadata`。

### 2.6 服务端日志也输出 request_id

之前管理页日志有 `request_id`，但控制台请求完成日志没有，客服仍然需要靠时间猜。

现在请求完成日志形如：

```text
POST /v1/chat/completions - 502 (3970 ms) request_id=178... error_code=upstream_payload_too_large
```

这样可以直接用同一个 ID 串：

1. 管理页日志。
2. 代理控制台日志。
3. 上游 new-api/sub2api 访问日志中的 `X-Request-ID`。

### 2.7 运行状态和 Windows 更新可观测性

新增/强化 `/api/runtime/status`：

- 当前版本、PID、GOOS/GOARCH。
- 配置路径、可执行文件路径。
- 监听 host/port、可打开的 admin URL。
- provider/model 数量。
- Windows 更新残留文件状态：
  - `.new`
  - `vs-ai-proxy-self-update.ps1`
  - `vs-ai-proxy-self-update.log`
  - `.bak-*` 数量和最近备份。

注意：

- 真实监听 host 仍保留为 `listen_host`。
- 如果 `HOST=0.0.0.0` 或 IPv6 通配，`admin_url` 展示为用户可打开的 loopback 地址，避免显示 `http://0.0.0.0:...` 这种不可用地址。

### 2.8 统一诊断摘要接口

新增 `/api/diagnostics/summary`：

- `runtime`：复用 runtime status 结构。
- `statistics`：总请求、成功/失败、平均耗时。
- `latest_failure`：当前保留日志中的最近失败。
- `provider_health`：provider 成功/失败、冷却、最近错误。
- `problem_providers`：连续失败、冷却中或有最近错误的 provider。
- `copy_summary`：可直接复制给客服的一段摘要。

目的：减少用户在健康页、日志页、更新状态之间来回跳转。

### 2.9 日志页增强

日志页调整点：

- “延迟”改为“耗时”，避免误解为纯上游延迟。
- 表格列更紧凑，给模型和工具/恢复列更多空间。
- `error_reason` / `error_action` / `diagnostic_summary` 前置展示。
- 最近失败卡片展示原因、处理动作、摘要、请求工具/响应工具样本。
- 过滤器支持：`q`、`provider`、`model`、`error_code`、`request_id`、`status_code`。
- 过滤在服务端执行，不是前端当前页过滤。
- 增加复制诊断摘要和复制最近失败摘要。

### 2.10 前端单文件漂移测试

`web/dist/index.html` 是单文件前端，容易出现：

- 新增按钮但 JS 引用 ID 写错。
- 复制旧卡片导致重复 DOM id。
- 事件绑定存在但 DOM 不存在。

新增 `web/index_html_test.go`：

- 检查 `id` 不重复。
- 检查脚本中所有 `$('id')` 引用都能在 HTML 中找到。

这不是完整前端测试，但能抓住单文件管理页最常见的发布级错误。

## 3. 工具调用门禁

### 3.1 为什么要新增门禁

v0.2.42 已经修复工具调用主链路，但如果后续发版只跑 `go test ./...`，仍然有两个风险：

1. 维护者不知道哪些测试是工具调用关键路径。
2. 快速热修时容易漏跑流式/DSML/legacy function_call 某个分支。

因此新增 `tests/tool_call_release_check.sh`，并接入：

- `make tool-check`
- GitHub Actions `check` job

### 3.2 `make tool-check` 覆盖范围

脚本分三段：

1. Provider JSON/tool fidelity：
   - `tools`
   - `tool_calls`
   - legacy `function_call`
   - object arguments
   - nested unknown fields
   - shell/git 参数保真

2. Proxy tool normalization / aliases / DSML：
   - 标准 OpenAI `tool_calls`
   - legacy `function_call`
   - 流式工具参数分片
   - DSML 方言
   - `run_tests` / `dotnet_test` / `npm_test` 等别名
   - `read_file` -> `get_file`
   - `write_file` -> `create_file`
   - `git_diff` -> `git`
   - 未声明工具不再被误改成 `<empty>` 或错误 block notice

3. Streaming business smoke：
   - OpenAI-compatible SSE 本地 mock 上游。
   - Ollama -> OpenAI SSE 本地 mock 上游。

通过标识：

```text
TOOL_CALL_RELEASE_CHECK_OK
```

### 3.3 工具调用当前可证明的结论

基于测试证据，可以证明：

- 代理不会在已知 OpenAI 标准工具协议里误删 `tool_calls`。
- 代理不会在 legacy `function_call` 路径里误删 `create_file/get_file`。
- 流式工具参数分片可以被下游还原为可执行 JSON 参数。
- DSML 方言在声明工具存在时可以转换为 OpenAI `tool_calls`。
- `run_tests` 这类常见 VS/Copilot 别名会映射到已声明的终端类工具。
- `create_file` 后接 `get_file` 的最小执行语义成立。

仍需谨慎的边界：

- 无法保证未来所有模型不会发明新的非标准工具方言。
- 无法保证真实 VS 本机权限、工具注册表、用户策略一定允许执行。
- 如果请求还没拿到模型响应就 413/499/503/timeout，工具调用自然不会产生。

## 4. v0.2.44 CI 采坑

### 4.1 现象

v0.2.43 tag 推送后，GitHub Actions 在 `make tool-check` 失败：

```text
== 3. Streaming business smoke ==
tests/streaming_test.sh: line 111: rtk: command not found
make: *** [Makefile:91: tool-check] Error 127
Error: Process completed with exit code 2.
```

### 4.2 根因

`tests/streaming_test.sh` 和 `tests/streaming_ollama_test.sh` 原本使用：

```bash
rtk go build -o "$PROXY_BIN" ./cmd/server
```

`rtk` 是本机开发环境中的包装命令，GitHub Actions 标准 runner 没有安装。

这属于典型的“本机脚本污染 CI”问题：

- 本地测试通过。
- CI 没有同样工具。
- 新增门禁后才暴露出来。

### 4.3 修复

将以下脚本统一改为标准 Go 工具链：

- `tests/streaming_test.sh`
- `tests/streaming_ollama_test.sh`
- `tests/useai_large_request_diagnostic.sh`

改为：

```bash
go build -o "$PROXY_BIN" ./cmd/server
```

### 4.4 为什么发布 v0.2.44 而不是覆盖 v0.2.43

`v0.2.43` tag 已经推送到 GitHub。

最佳实践是：

- 不重写已经推送的 release tag。
- 不强推覆盖用户可能已经看到的 tag。
- 发布补丁 tag `v0.2.44` 修复 CI/发布流程问题。

因此 v0.2.44 是一个发布流程修复版本，功能代码相对 v0.2.43 只改测试脚本依赖。

## 7. 后续维护原则

1. 不要再把本机专属命令写进 CI 会跑的脚本，例如 `rtk`。
2. 工具调用变更必须同时考虑四类协议：标准非流式、legacy 非流式、标准流式、legacy 流式。
3. DSML 属于文本方言转换，必须继续保持“只转换已声明工具”的边界。
4. OpenAI 标准工具调用默认稳定透传，不要恢复旧的“未声明就删除”策略。
5. `request_id`、未来 `trace_id` 等请求级元数据必须走 `internal/requestmeta`，不要在 proxy/provider 分散新 context key。
6. 管理页单文件变更必须跑 `go test ./web -count=1`，防止 DOM id 和 JS 引用漂移。
7. 诊断字段新增时必须同步考虑：Store 日志、API JSON、前端展示、复制摘要、文档和测试。

## 4. 旧 session 失败、新 session 恢复的解释口径

最近用户反馈了一个非常关键的现象：

- 同一模型在旧 session 中连续失败，表现为 `499 client_gone`、`502 network_error`、`context canceled`。
- 重新开一个新 session 后，模型又恢复正常。

这个现象不能简单地解释成“模型本身坏了”或“代理全局坏了”。更符合工程经验的解释通常是：

1. 旧 session 中积累的历史消息、文件内容、工具调用记录、重试痕迹过多，导致请求体膨胀。
2. 请求体膨胀后，上游首 token 变慢、网关处理变慢、连接更容易被客户端等待上限截断。
3. 当新建 session 后，请求体显著变小，链路恢复可用，因此看起来“只有新 session 才正常”。

因此，日志上的推荐口径应当是：

- 如果失败请求的 `request_bytes` / `upstream_bytes` 偏大，且新 session 后恢复，**优先怀疑旧 session 上下文膨胀或状态污染**。
- 这类问题不应仅归因于 `network_error` 或 `client_gone`，而应结合请求体大小、耗时、流状态、远端 peer 一起看。
- 若同一 session 下从 `499` / `502` / `context canceled` 转为新 session 成功，通常说明“会话内容压力”是一个强信号，而不是唯一根因。

### 4.1 本轮日志增强后的输出效果

当请求体或上游体达到大上下文阈值时，日志会在 `error_action` / `diagnostic_summary` 中补充类似提示：

- `本次请求体/上游体约 619.2 KB，属于大上下文；如果新建 session 后恢复，优先怀疑旧 session 历史膨胀、文件堆积或状态污染。`
- `本次请求体/上游体约 1.03 MB，属于超大上下文；如果新建 session 后恢复，优先怀疑旧 session 历史膨胀、文件堆积或状态污染。`

这样客服和维护人员不需要再猜“是不是网络”“是不是 nginx”“是不是 new-api 故障”，而是能先判断是不是会话压力导致的链路退化。

### 4.2 需要继续观察的边界

这仍然不是绝对结论。下面几类情况都可能与旧 session 恢复现象同时出现：

- `upstream_payload_too_large`：更偏向明确的大请求拒绝。
- `client_deadline_reached`：更偏向 VS/Copilot 等待上限触发。
- `network_error` / `use of closed network connection`：可能是上游慢、链路断开、CDN/WAF、中间代理关闭，也可能是大请求把窗口拉长后更容易暴露出来。
- `proxy_parse_error`：如果上游把 SSE/HTML/非 JSON 混着返回，也会伪装成“模型不行”。

因此，本轮日志增强的目标是“把最可能的方向第一时间写出来”，不是伪造确定性。

## 5. 后端诊断到前端页面的闭环核查

本轮复查确认，核心诊断链路已经形成闭环：

1. `internal/proxy/diagnostic_error.go` 负责把底层错误归类为稳定的 `error_code`，并生成 `error_reason`、`error_action`、`diagnostic_summary`、`attempts_summary`。
2. `internal/proxy/server.go` 的 logging middleware 会把 provider、model、upstream、请求体大小、上游体大小、流状态、远端 peer、取消原因、工具声明等字段写入 `store.RequestLog`。
3. `internal/store/store.go` 保存完整日志，并提供分页查询、过滤、最近失败样本和统计信息。
4. `internal/api/api.go` 通过 `/logs` 和 `/diagnostics/summary` 暴露这些字段，`copy_summary` 可直接复制给客服或维护人员。
5. `web/dist/index.html` 的日志页会展示：
   - 表格中的错误徽标：`error_code · error_reason`。
   - tooltip 中的原因、处理建议、诊断摘要、候选耗时、请求体/上游体、超时、远端 peer、流状态。
   - 最近失败卡片：优先读取全局 latest failure，而不是只看当前分页。
   - 复制诊断摘要/最近失败摘要按钮：用于客服快速收集上下文。

因此，当前闭环是：

```text
上游/客户端错误 -> proxy 分类和结构化诊断 -> store 持久化 -> API 查询/摘要 -> Web 日志页展示/复制
```

### 5.1 当前闭环能解决的问题

- 多候选 provider 失败时，不再只看最后一个错误，而是展示最可行动主因。
- `API 错误 400/413 ... context canceled` 不会误判成纯客户端断开。
- 旧 session 请求体偏大且新 session 恢复时，日志会提示优先排查上下文膨胀/文件堆积/状态污染。
- 413 提示会指向真正触发大请求错误的 provider/model，避免排错对象张冠李戴。
- 前端能直接看到请求 ID、provider、model、upstream、耗时、工具声明、流状态和处理建议。

### 5.2 尚需谨慎的边界

- 日志能给出“证据型推断”和“优先排查方向”，不能凭单条日志证明上游内部某个渠道一定故障。
- `network_error` 仍可能同时包含本机网络、CDN/WAF、源站连接、上游慢首 token、大上下文放大等因素，需要结合 `request_id` 到上游网关日志继续核对。
- 前端当前展示已经可用，但还不是完整 APM；如需更强闭环，后续可增加按 request_id 展开的单请求时间线。


### 5.3 本轮继续深挖发现的闭环断点

复查发现后端已经支持 `error_reason` 过滤，但日志页筛选栏没有对应输入框，导致客服只能通过通用搜索间接过滤原因，闭环不够直观。

已修复：

- 日志页新增 `error_reason` 输入框。
- 前端 `getLogFiltersFromUI()` / `setLogFiltersToUI()` 补齐 `error_reason` 参数。
- `/api/logs` 过滤测试增加 `error_reason=上游服务`，确保 Web 参数、API 参数和 Store 过滤字段保持一致。

注意：`q` 仍然是全文模糊搜索，`error_reason` 是专用原因筛选；客服排障时优先使用专用字段，减少误匹配。

### 5.4 深度复审补充：分类正确不等于控制流正确

进一步复审发现一个更隐蔽的风险：即使 `classifyProxyErrorFromErr()` 已经能把 `API 错误 500 ... context canceled` 归类为 `upstream_server_error`，流式请求主流程仍可能通过 `isClientGoneError()` 只看 `errors.Is(context.Canceled)`，从而把带明确上游 5xx 的错误当成客户端断开直接返回。

这类问题的危害是：

- 日志层看起来像 `client_gone` 或没有完整诊断。
- 真实上游 5xx 被吞掉，客服无法看到 `upstream_server_error`。
- 多候选 provider 的失败切换和错误摘要可能被提前中断。

已修复：

- `isClientGoneError()` 先调用统一分类器，只有最终分类确实是 `client_gone` 时才按客户端断开处理。
- 新增集成测试 `TestStreamingClassifiesUpstreamStatusBeforeContextCanceled`，确保带 `API 错误 500` 的混合取消不会被吞掉。
- 新增矩阵测试 `TestClassifyProxyErrorPrioritizesExplicitHTTPStatusOverTransportText`，覆盖 401/403/400/404/413/429/500/503 与 `context canceled`、`timeout`、`client_gone` 等传输文本混杂的场景。

复审结论：今后修改错误分类时，必须同时检查“日志分类”和“主流程控制分支”，不能只看最终展示字段。
